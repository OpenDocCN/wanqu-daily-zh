# 活在科技写作的未来

*Pro Git 的惊人冒险和最终工具链，第二版*

差不多就在六年前，一家出版社找到我，请我帮忙写一本名为《Pro Git》的书，这本书正在写作中，但进度落后了。最终原作者决定不再写作，我从头开始重写这本书，最终在 2009 年 8 月出版。对于前三章左右，我用 Word 写了这本书，将文档发送给我的编辑，过一段时间后取回红线版本。

读了大约三章后，当我建议我们在写作和技术编辑阶段切换到 Markdown，然后回到 Word 进行文字编辑时，我几乎要放弃了，我同意了。这本书完成后，我再次将所有内容移回 Markdown，这样我就可以在自己创建的网站上在线发布了。对我来说幸运的是，原作者已经同意了知识共享许可协议。

一年后，Apress 也发布了这本书的 Kindle 版本，尽管这本书一发布，人们就已经从开源内容中生产出了 Kindle 兼容版本。

# Pro Git，第二版

由于这本书卖得相当好，在印刷了大约四年之后，一家出版社找到我，让我再版这本书。在那段时间里，我的公司 GitHub 已经从一个为几千用户服务的网站的四个人成长为一个拥有数百万个知识库和 200 多名员工的巨大的在线存在。不仅 GitHub 社区发生了如此大的变化，而且这个工具本身也发生了一些重大变化。我同意可能是时候修改内容了。

虽然你现在可以在线阅读第二版(以及第一版)或者下载 PDF、ePub 或 Mobi (Kindle)格式的版本，以便在你最喜欢的设备上阅读，但我认为可以分享的一个有趣的故事是，这个版本与大约五年前的第一版在制作方式上有所不同。

# Pro Git v2 的编写

事实上，在过去几年中，我已经开始对 Pro Git 进行了几次改进。我主要对它的工具链方面感兴趣。在某个时候，我开始了一个名为 [Git Scribe](https://github.com/schacon/git-scribe) 的项目，该项目旨在展示我*希望*能够用来编写 Pro Git 的工具链类型。scribe 工具提供了一种半简单的方法来安装工具链，该工具链将采用特定格式的 Asciidoc 书籍，并从中生成好看的 HTML、PDF、ePub 和 Mobi 电子书。虽然这个项目后来夭折了，我们最终改用了 O'Reilly 的 Atlas 平台，但 Scribe 项目首先让我进入了 Asciidoc。

## Asciidoc

所以我把 Pro Git 的很大一部分换成了 Asciidoc。降价的问题在于它太简单了。它没有指定表格格式、交叉引用、索引、标注、源代码示例等。所有这些都是 Asciidoc 以同样易于编写的格式完成的。

所以我把 Markdown 的所有内容都转移到了 Asciidoc，并注意到一些出版商也在这么做。O'Reilly 开始允许用 Asciidoc 编写书籍，因为它很容易生成 Docbook，我相信务实的程序员在此之前已经使用简单的标记语言有一段时间了。

最重要的是 Asciidoctor 的兴起。当我去重做我的 Git 网站(git-scm.com)时，我想获取 Git 的 Asciidoc 手册页源代码，并制作它们的在线版本，我想用 Ruby 来做。我的朋友兼 GitHub 同事 Nick Hengeveld 为 Asciidoc 编写了一个很好的解析器和格式化程序来帮助完成这项工作。最终，其他一些 githubber(Ryan wald Ron 和 Jeremy McAnally)加入进来，将 Asciidoc.rb 变成了 asciidor，最终由令人惊叹的 Dan Allen 接管，他将该项目提升到了一个全新的水平。

现在有了 Asciidoc，我可以用我的 Asciidoc 源代码书做一些真正令人惊奇的事情，这些事情远没有以前那么简单。

## 开源代码库

编写工作流程的最大变化之一是 GitHub 现在比以前好了很多。现在，我们有了用于审查和评论的 Pull 请求、用于章节规划的问题里程碑、用于更容易审查和抽查的散文差异，以及用于写作和 Asciidoc 预览的 Atom 文本编辑器。



我们必须使用的工作流程是每一个曾经写过技术书籍的人，尤其是和合著者，可能会非常嫉妒的。

我们写下了所有我们想做的事情，将所有要点作为问题添加到每章的里程碑上，然后每个人决定我们要做什么，并开始写作。我们可以为我们正在做的工作单元创建一个分支(通常是一章),然后开始编写并推送到 GitHub 分支。我们聊得很轻松，每当我们中的一个人推出一些变化时，我们都会得到即时更新。我们几乎立即打开了一个 Pull 请求，通常带有一个在合并之前我们想要完成的事情的清单，这样您就可以知道每个分支的状态了([示例](https://github.com/progit/progit2/pull/89))。

当一个分支运行很长时间时，我们可以将主分支合并到其中，使其保持最新，并确保冲突永远不会成为一个大问题，即使是通过大规模的结构变化，如将章节分成多个文件。

审查是通过拉式请求完成的——我们可以在任何一行介绍的文本上留下评论，并在那里或在聊天中就此进行对话。

我能够安装 [Asciidoc Atom](https://github.com/asciidoctor/atom-asciidoc-preview) 预览插件，并在我写的时候获得我正在写的文本的实时渲染视图，包括关于我添加、修改或删除了什么的侧边栏提示，我在哪个分支上，合并冲突帮助工具，zen 模式等。这是一个令人惊叹的写作环境。



不仅如此，我们大部分的合作都是在相隔 9 个时区的时候完成的，我们的编辑可以随时查看里程碑页面来了解我们在这本书上的进展。

如果你写了一本技术书，但没有在现代 GitHub 上用 Asciidoc 写，那么你已经做了十倍于你需要做的工作。

## 阿特拉斯

这本书不仅比第一版更容易写作和合作，而且看起来也更容易。



The O’Reilly Atlas build system. Point and click for beautiful ebooks in minutes.



几年前，我与蒂姆·奥莱利(Tim O'Reilly)、劳拉·鲍德温(Laura Baldwin)和奥莱利媒体(O'Reilly Media)的其他人讨论过这个工具链可以有多好，这在一定程度上归功于我的对话，我受邀试用他们正在开发的平台，以使这一切变得更容易。我们都同意从 Asciidoc 到‘git push’再到 PDF 会有多好。他们最终真的建造了它，谢天谢地，他们让我试了试。

你可以看到我在他们的论坛上活跃了几个月，每次我提到什么，他们都会在几天甚至几小时内解决。不仅如此，构建产生了如此漂亮的输出，以至于对于章节回顾，我最终构建了一个 PDF，下载它并在预览中阅读，同时在 Asciidoc 源代码中做笔记和修改。

幸运的是，他们也有服务的 API，你可以分别构建不同的分支。这意味着我能够编写一个服务来监听 Pro Git 存储库上的 post-receive 挂钩，提取更改，将它们推送到 Atlas 并自动开始构建。然后，它会轮询所有构建何时完成，并将它们移动到我自己的 AWS S3 桶中，并更新新的 progit.org 网站和 git-scm.com 网站的内容和链接。

不仅如此，我还可以为 Pro Git 翻译成的每一种语言做这件事。这意味着，在不久的将来，如果你去 git-scm.com 的[或](https://progit.org)[网站，你可以下载最新版本 Pro Git 的任何格式的专业级电子书。在任何语言中。点击“合并”按钮后，从我或语言维护人员那里得到零。](http://git-scm.com)

我现在基本上拥有了这个星球上最简单和最复杂的多语言电子书制作和发行系统。

## 开放源码

第一版和第二版之间的另一个有趣的区别是，因为我不想再经历一次从 Word 到 Markdown 的过程，我们同意我可以完整地写这本书，然后“把它扔到墙上”给出版社，出版社会进行 Word 转换和文字编辑，然后把修改发给我们。这意味着在这个过程中，本和我能够比以前更早地开放这本书的源代码。

这本书目前在网上完全可以买到，包括所有的电子书格式，但是可能离印刷还有几个月的时间。在这段时间里，我们肯定会得到大量的小的文字编辑和技术修改(在最初的 24 小时里我们已经得到了一些)，并且能够在印刷之前将它们合并。这意味着我们可以在实际印刷这本书之前找到并修复勘误表，这是相当惊人的。

实际上，我们讨论过是公开写完整本书，还是等到它基本完成后再重新打开。最终，我们选择了后者，尽管我仍然不确定前者是不是更好的选择。我们觉得在整个过程中处理来自整个社区的意见可能太麻烦了，我们更喜欢从根本上拥有内容，但走另一条路可能会很有趣。也许下次吧。

## 翻译

这一次最大的不同是对作品翻译的处理。当开源第一版的时候，我完全不记得有过翻译的想法。我不记得得到我的第一个翻译作品，但它必须在我发布后相当快，因为我在发布后不久就重组了目录，将所有内容放在一个“en”目录中。



在这本书出版的五年里，它被完整地翻译成了至少 10 种语言，简体中文,正體中文，法国人，日本語、荷兰、русский、한국어、葡萄牙语和埃什蒂纳语)并部分翻译成大约另外 20 种语言。因为我决定将它们存储在主存储库的子目录中，并接受对它们的拉请求，这意味着对存储库有写访问权限的人必须合并它们。这意味着我基本上不得不合并所有的东西，尽管我几乎看不懂这些语言中的任何一种。最终，几年后，我变得很难跟上拉取请求，令人惊叹的[让-诺尔·阿维拉](https://github.com/jnavila)接管了我的工作，并从那时起开始管理翻译(甚至是英语勘误表)。

不过，这一次，我们可以从以前的问题中吸取教训，努力让大家都轻松一些。这一次，我们将每一个翻译作为 progit 组织下的一个单独的库，并为每个库添加特定语言的维护者。这意味着你可以用中文在 ZH 版上开一期，而不会让我们完全困惑。如果维护者不再响应，而其他人感兴趣，我们可以将他们添加为新的维护者。

多亏了 Atlas，每个翻译都将有自己的自动化构建过程，而不是像现在这样让每个人都在本地运行自己的构建。

## 概括起来

我相信这个过程至少是技术出版的中短期未来。以更简单的格式写书，如 GitHub 上的 Asciidoc，轻松地与编辑和合著者协作，开源内容，拥抱强大的翻译社区，以及使用 Asciidoc 或 Atlas 等工具自助出版。

我还认为，所有这些对整个出版业都有非常严肃和有趣的影响，我将在以后的文章中深入探讨。
