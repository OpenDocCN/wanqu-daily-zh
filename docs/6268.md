# 一次性代码

> 原文：<https://blog.bencope.land/throwaway-code/?utm_source=wanqu.co&utm_campaign=Wanqu+Daily&utm_medium=website>



有时我们会编写一些我们知道会被丢弃的代码——尤其是在一个特性或产品的发现阶段。问题是，通常情况下，这种代码会被发布。我可以想象，从事代码工作的前一位开发人员可能会因为这种情况而受到相当多的诅咒。不过，编写一次性代码并不一定是件坏事。在画家的画布上，最终作品的下面可能有许多层，就像一部完成的小说或剧本被丢弃或重写了整个部分一样。我们想要避免的是，一旦决策者看到“它正在工作”，不可维护的代码(故意这样写的)就被迫退出

在他的书*游戏编程模式*中，Robert Nystrom 就如何在构建原型的情况下防范这种情况提出了一个很好的建议。

> 确保你的原型代码不会变成真正的代码的一个技巧是用不同于你的游戏使用的语言来写它。那样的话，你必须重写它才能在你的游戏中结束。

不管使用什么方法，他指出，我们应该“确保使用一次性代码的人明白，即使它看起来可以工作，但它不能被维护，T2 必须重写”
