新闻稿(PR)常见问题(FAQ)是一个以客户为中心的文档，用于设计新产品。它是一个理想化的未来“新闻稿”(PR)和相关的常见问题。PR FAQ 是您其他产品文档的起点。

这篇文章是写给任何对公共关系常见问题处理感兴趣的人的。公共关系常见问题的一个优势是，组织中的任何人都可以理解它们。因此，这篇文章是写给那些正在考虑采用 PR FAQ 产品文档的产品经理的，但是它也同样适用于任何参与构建技术产品的人。

公共关系常见问题遵循相当严格的格式:

1.  **一篇新闻稿** (PR)，从未来新产品发布时的角度写的。
2.  **公共常见问题解答**，详细说明客户可能对产品有什么疑问，就好像是与 PR 同时发布的公共产品文档一样。
3.  **内部常见问题解答**，回答内部利益相关者在产品开发过程中提出的任何问题。



PR FAQ 格式已经存在一段时间了，但它是由亚马逊(我第一次在那里学到它)推广的，现在被产品经理广泛使用。我使用的产品格式和审核流程与亚马逊略有不同，自从我在亚马逊工作以来，我已经对其进行了改进。我将在下面详细介绍这种格式，并在另外两篇文章中讨论审核流程:

本文是系列文章的一部分，主要关注 PR FAQ 文档本身。

# 基本公关常见问题格式

一份新闻稿通常有一到一页半的篇幅，遵循一种非常模板化的格式，你将在下面看到。在亚马逊，这一点被严格执行。像一个新的搜索栏这样的小功能可能是一页的 PR，不会更短。以 137 亿美元收购 WholeFoods 可能只是一页半的公关，但现在不再是了。

FAQ 是开放式的，可以持续多长时间，这是简单产品和复杂产品之间最大的区别。例如，一个简单的功能可能有 2 页的 FAQ，而一个更复杂的产品可能有 20 页的 FAQ。

公共关系常见问题应该是所有其他产品文档的起点:

*   您的工程团队应该能够使用您的 PR FAQ 作为范围界定的起点；您的设计团队应该能够使用您的 PR FAQ 作为设计文档的起点
*   销售团队应该能够利用你的公共关系常见问题来设计新产品的销售行动手册
*   营销应该能够使用公共关系常见问题来创建实际的新闻稿
*   你的执行发起人应该能够利用你的公共关系常见问题来了解构建产品所需的资源分配，等等。

公关常见问题是一个很好的门控功能。如果你不能创造一个所有利益相关者都清楚的公关，那么这可能是一个糟糕的产品创意:也许你在一个产品中塞进了太多东西，或者产品的影响没有解决真正的痛点。如果利益相关者问了你在 FAQ 中无法回答的问题，那么你还没有对产品投入足够的思考来开始构建它。

# 公共关系常见问题模板

这里有一个好的公关常见问题的模板。纯文本概括了每一部分的功能，斜体文本强调了每一部分试图实现的功能和测试的假设。

## 新闻稿

## **标题:目标客户能够理解的产品简称**

**小标题:一句话说明谁是市场，什么是利益**

总结:用 2-4 句话概括产品和优点。应该是独立的，这样人们可以只阅读这一段，仍然理解新产品/功能。

问题:用 2-4 句话描述客户面临的问题，该产品可以解决这些问题。*测试您对正在解决的棘手问题的假设。*

解决方案:2-4 句话，描述新产品/功能如何解决这个问题。测试你对如何解决棘手问题的假设。

入门:用 1-3 句话描述某人如何开始使用该产品/功能(如果它已融入现有产品，请明确说明)。*测试您对客户利用新产品/功能的难易程度的假设。*

内部引用:您公司内部有人被引用了他们对产品/功能的喜好。测试您对自己为客户创造的价值的假设，以及您如何在更广泛的产品中定位该产品。

客户报价:一个假设的客户说他们喜欢新产品/功能的什么。测试你关于你希望客户对新产品/功能的反应的假设，以及你的理想客户档案。他们应该做一些他们以前不能做的事情，做一些更快更容易的事情，节省时间和精力，或者以其他方式让他们的生活变得更好。无论好处是什么，他们对好处的喜悦应该在报价中表现出来。如果您有多个理想客户，例如:中端市场和 F50 客户，这应该是来自不同客户的多个报价。

行动号召:用 1-2 句话告诉读者下一步可以从哪里开始使用产品/功能。测试你的假设，这是否是一个自动开启的功能，他们需要开启的东西，测试版等等。

## 常见问题

一组公众常见问题及其答案。这应该是一个客户可能想知道的关于产品的所有信息的综合列表。它应该包括在产品/功能开发期间与客户和面向客户的团队讨论新产品/功能时出现的任何合理的问题。

## 内部常见问题

一组私人的、内部的常见问题及其答案，采用其他所有利益相关者都能理解的格式。常见问题可能包括具有强大 UX 组件的产品的线框，或指向单独线框文档的链接，但 PR 应仅依赖于文本。这将允许所有内部利益相关者清楚地了解产品/功能。



# 为什么 PR FAQ 格式有效？

PR/FAQ 格式有效，因为它:

1.  以客户为中心
2.  迫使你的假设变得清晰
3.  任何利益相关者都可以解释
4.  增加利益相关者的所有权
5.  简单到任何利益相关者都可以创建

## 1.以客户为中心

PR/FAQ 策略之所以有效，是因为它是以客户为中心的，通常被称为“逆向工作”，因为您是从发布开始，并回填细节以达到目的。将最终产品的愿景记在心里始终是项目经理的工作，没有什么比将最终产品作为您的第一个文档更有帮助的了。

有几种方法可以让客户影响在 PR FAQ 中更加突出。第一种方法是使用客户报价。正如上面的 PR FAQ 模板中所说，您可以有多个客户报价来捕捉您将要进入的不同市场，您也可能希望有多个客户报价来捕捉您的产品将影响您的客户的不同方式。让客户影响更加突出的第二个方法是在你的常见问题中明确说明。例如，我总是包括“我们的客户最喜欢产品的什么？”作为内部常见问题的第一个问题。因此，即使是公司内部的讨论也是以客户为中心的。

## 2.明确你的假设

PR FAQ 帮助你测试关于客户驱动的特性是否真的会产生很大影响的假设。例如，如果你写了你的第一个 PR FAQ，而你正在解决的痛点是每天“额外的鼠标点击”,那么它就测试了你关于这个特性有多重要的假设。你的客户可能真的讨厌额外的鼠标点击带来的冗余感，你可以用一个解决这个问题的新设计让他们惊叹，但是你可能用同样的资源解决更大的问题。

因此，公共关系常见问题应该包含足够的信息，让人们讨论为什么这个特定的产品/功能是最重要的下一个功能。请确保将关于产品范围的讨论与关于产品在路线图上的优先级的讨论分开:它们是与不同的利益相关者进行的不同讨论，试图同时解决范围和优先级可能会太混乱。

## 3.任何利益相关者都可以解释

PR/FAQ 文档是一个统一的风格，所有利益相关者都应该能够理解。在 AWS，我必须让大约 50 个不同的人签署我的产品:销售、法律、设计、本地化、营销、多个工程团队等。在我与每一个人交谈的开始，他们都询问了公共关系常见问题。这真的让我意识到 PR FAQ 过程是多么有效:每个人都希望产品文档的格式是他们非常熟悉的，并且可以立即适应他们的任务。如果这 50 份文件中没有不符合期望的差异，就不可能使文件适应 50 个不同的利益相关者。

特别是文档中的 FAQ 组件，在以一种所有其他利益相关者都能理解的一致格式捕捉不同利益相关者的假设和观点方面极其有效。

## 4.增加利益相关者的所有权

事实上，任何人都可以添加到一个公共关系常见问题是一个力量，以确保买入。每个人现在都可以感受到对产品及其成功的承诺。一个人给出的任何反馈都可以立即作为一个问题添加到 FAQ 中。这是确保人们的声音被听到的一个很好的方法:他们可以马上在产品文档中看到他们的真实话语。每个人的反馈都是平等的:一个实习生的问题看起来和一个高管的问题一样；来自技术含量最高的利益相关者的问题看起来和来自技术含量最低的利益相关者的问题一样。

相比之下，并不是所有对抽象表达或线框的反馈都是相同的。更熟悉产品管理的人会更好地理解约定，并能够给出更详细的反馈。与将他们的问题添加到一个常见问题相比，其他利益相关者不会看到他们的反馈被采纳或了解他们的反馈将如何被立即执行。

作为产品经理，你的职责是确保每个人的声音都被听到，所以你不希望产品文档的格式成为一个门控功能。如果有人对线框或“幻灯片”给出反馈，他们可能不清楚他们的反馈是如何在下一次设计迭代中被考虑的。当他们将自己的问题视为 FAQ 的一部分时，就没有这种模糊性，因为这是他们自己的话，即使 FAQ 问题的答案不是他们期望或想要听到的。

## 5.可以由任何利益相关者创建

组织中的任何人都应该能够编写公共关系常见问题，而不仅仅是产品经理。这是这种格式的优势之一:你不需要成为一个项目经理或者知道任何特定的软件来创建产品文档。很可能项目经理仍然会指导产品开发过程，但是第一个想法可以来自任何人。

对于 AWS 最近的一个机器学习产品，第一个公共关系常见问题是由科学团队的一名实习生编写的。这个人最终成为了产品团队的全职员工，以见证新产品上市的整个过程。因此，如果你是一个喜欢从你的组织的其他部分开发良好的项目管理的产品领导者(老实说，每个好的产品领导者都应该这样做)，那么公共关系常见问题是一个很好的方式，让人们测试他们有多喜欢产品管理。

我获得的产品领导经验越多，我就越相信产品的作用是促进产品开发过程，而不是定义产品和功能应该是什么样子。产品经理可能会想出前所未有的天才的新产品创意，但不会比公司其他职位的人想得更多。如果你*不能*让其他人从和你一样的数据中独立地提出同样的产品想法，那么也许它一开始就不是一个天才的想法。因此，PR/FAQ 为组织中的每个人提供了一个表达他们最初产品想法的框架，从而创造了一个公平的竞争环境。它打开了通向更多想法的漏斗，并鼓励最佳想法获胜的精神，不管它来自谁。

# 这个 PR FAQ 格式和亚马逊有什么区别？

我的公关常见问题文档与亚马逊的有三个主要不同之处:

1.  允许一些设计元素
2.  提供更多关于潜在解决方案的详细信息
3.  以多种不同的方式获得反馈

## 1.在你的常见问题中允许一些设计元素

亚马逊的产品流程以文本驱动著称。我在那里的时候，除了对外演讲，从来没有看过 PowerPoint 演示。与我一起工作的 UX 设计师是有史以来最强的设计师之一，但相对于我工作过的其他地方，设计进入产品过程较晚。

没有必要像亚马逊那样关注文本，关注如何运行公共关系常见问题解答流程。我同意 PR 应该保持为文本:从早期的图形设计中很容易有未言明的假设，所以要求 PR 为文本是一个很大的强制功能，以使假设明确。但是 UX 设计可以在早期提供帮助。在我今天的角色中，我们正在为人类和机器学习之间的实时交互建立技术。人机交互，或者说 UI 设计，对我们的产品来说非常重要。所以，我的妥协是在内部 FAQ 中允许一些图表(不在 PR 中)。这可能非常简单，可以被框定为一个问题:

问:该产品 MVP 的线框是什么样的？

我包括 1 或 2 张图片，最多，并链接到设计文件的更多细节。

对于评审会议，不要让这些设计元素成为人们解释产品的出发点:这正是基于文本的方法试图避免的。当在 PR FAQ 中引起对设计元素的注意时，我建议询问人们这些设计是否反映了他们对 PR FAQ 本身的理解。这就保持了对公共关系常见问题的关注，作为产品内容的真实来源，而不是让视觉元素成为真实的来源，否则会发生在更注重视觉的人身上。

## 2.提供更多关于潜在解决方案的详细信息

您的工程团队可能会立即开始考虑产品的解决方案:什么是代码库、数据库、缓存解决方案等等。？

与设计元素一样，这些元素可以作为问题添加到常见问题解答中，例如:

**问:我们现有的数据库解决方案能否支持这一新产品？**

虽然产品文档应该独立于技术解决方案，但它可以帮助包含一些细节，让审阅者了解潜在解决方案的规模。在较大的公司里，这可能无关紧要，但特别是如果你在一家预算有限的初创公司，你的主管很可能希望在产品设计过程的早期就了解所需的资源。我建议所有答案都是建议:“*我们可能……*”、“*有可能……*”等。，这样就不会混淆这些是实际的技术规范，或者在这个阶段任何东西都是一成不变的。您可能会无意中误导工程师，让他认为技术解决方案已经确定，而这只是帮助确定范围的早期推测。

在确定了一些潜在解决方案的范围并规划了资源投入后，高层次的细节应该进入常见问题解答。但是，它们应该链接到范围界定文档，不要让任何人误以为 PR/FAQ 是这一级别的工程细节的基础。出于组织原因，这一点很重要:您的工程团队应该感觉他们拥有范围界定文档，就像您的销售团队应该感觉他们拥有关于产品的剧本材料，而您的营销团队应该拥有实际的 PR，等等。没有人希望感觉产品经理在微观管理他们的工作——这是一种常见的(mic)看法——所以清楚地划分产品文档的所有权是了解谁拥有什么的一个重要组成部分。

## 3.以多种不同的方式获得反馈。

亚马逊的公共关系常见问题流程就像文档本身一样结构化。我将在接下来的两篇文章中介绍审核流程和确保反馈多样性的方法:





# 公关常见问题示例

这是一个虚构产品的公共关系常见问题的例子。这是一种预测电子邮件功能，试图预测某人在写电子邮件时的下一句话，就像你可能已经看到一些公司最近发布的那样。

请记住，公关常见问题是一个不断编辑的活文档——我在大公司见过 100 个左右的版本号。把这个例子想象成版本 4 或 5:一个项目经理已经通过了几次审核，并且与他们的直接经理进行了第一次评审，因此公关已经准备好与内部利益相关者进行第一次会议。

# **Acme 向其客户发布智能电子邮件**

***Acme 的智能邮件工具使用预测文本，使写邮件的速度提高 10 倍***

*2019 年 8 月 4 日*。Acme 很高兴地宣布，我们的电子邮件产品将添加智能预测文本。新功能“智能电子邮件”可以在客户键入电子邮件时提示句子中的下几个单词。客户可以按 Enter 或 Tab 键来接受这些单词，而不是手动键入每个单词。因为我们写的许多电子邮件往往有许多重复的文本，这可以使撰写电子邮件的速度提高 10 倍。使用 Acme 智能电子邮件，您可以专注于电子邮件中独特和重要的内容，将机械的部分留给智能建议。

写电子邮件是一项重复的任务，在一天中，许多人一遍又一遍地键入一些短语，如“请查看附件”，或“让我们下周安排一次会议”。具有首字母缩写词和模板的现有解决方案具有严重的缺点。首字母缩略词会造成混乱，或者让电子邮件显得匆忙和不礼貌。许多人使用电子邮件模板，但模板编辑起来可能效率低下:当有人在几个不同的地方点击模板来更改文本时，从头开始手动键入整个电子邮件可能会更快更容易。

智能电子邮件通过消除打字中的冗余，有效地解决了创建长格式电子邮件的问题。如果你开始键入一个句子，如“请查看……”，那么智能电子邮件可能会建议“……附件”，你只需要按一个键就可以将这些单词添加到你的电子邮件中。智能邮件也会学习你的风格。例如，如果你用“最好”或“亲切的问候”来结束电子邮件，或者如果你拼写为“颜色”或“色彩”，智能电子邮件将开始学习你的风格偏好并适应这些风格。

Acme 电子邮件用户今天就可以开始使用该功能。当您开始在浏览器或我们的 iOS 和 Android 应用程序中撰写电子邮件时，您现在会看到对您可能想要键入的下几个单词的建议。

**常见问题解答**

**1。如何开始使用 Acme 智能电子邮件？**

如果您在[LINK]登录您的 Acme 电子邮件帐户，或者在您的手机上打开 Acme 电子邮件应用程序，当您开始键入电子邮件时，您应该会立即看到智能建议。

**2。我必须使用来自 Acme 智能电子邮件的建议吗？**

来自 Acme 智能电子邮件的建议是可选的:您可以忽略它们，键入您喜欢的任何内容。您也可以在设置->智能设置中完全关闭 Acme 智能电子邮件。

**3。智能建议也适用于 Acme 文档和 Acme 工作表吗？**

不，现在智能建议仅限于 Acme 电子邮件。

**4。Acme 智能电子邮件支持哪些语言？**

如今，Acme 智能电子邮件支持英语、西班牙语、印地语、阿拉伯语、中文(简体)、Matsés 和德语。我们将在未来添加更多的语言，并欢迎关于下一步支持什么语言的建议！

**5。Acme 智能电子邮件如何从我的电子邮件中学习？**

智能邮件使用最先进的机器学习模型来适应你的风格，并学习如何为你写一些句子。

当你输入几个相同的单词序列时，比如“请查看附件”，那么智能电子邮件可以学习到“请查看附件…”后面经常跟着“…文档”。智能邮件从大量现成的这类知识开始，然后适应你的风格。如果你经常键入“请看附件的 ***演示文稿*** ”，那么 Acme Smart Email 就会学习这个模式，开始建议“…演示文稿”而不是“…文档”来完成你的句子。所以，它会变得越来越聪明，你用它！

**6。你会用我的邮件给其他顾客提建议吗？**

不，您的数据安全是我们的首要任务。我们不会与任何人分享您的电子邮件内容。为您定制的预测文本功能也只对您可用:我们不会根据我们从您那里了解到的信息来改变其他人的建议。

**7。Acme 智能邮件需要多长时间来学习我的风格？**

通常需要 1000 封左右的电子邮件，您才能从 Acme 智能电子邮件中获得最佳收益。我们的客户立即开始看到 2 到 5 倍的速度提升，但是直到写了大约 1，000 封电子邮件之后，您才可能看到我们的许多客户所享受的 10 倍的速度提升。

对此没有时间限制。如果你通常一天发送 1000 封电子邮件，那么你应该会在一天内看到非常大的速度提升！

**内部常见问题解答**

**1。客户最喜欢 Acme 智能电子邮件的什么？**

客户会喜欢它给他们带来的效率提升，以及删除写电子邮件中最多余部分的普遍快乐。

**2。客户最不喜欢 Acme 智能电子邮件的什么？**

客户会对语言覆盖感到失望。特别是，我们在拉丁美洲有许多同时使用西班牙语和葡萄牙语的客户，但是我们在第一版中只支持西班牙语。我们将亲自提前联系这些客户，确认缺少葡萄牙语，这是产品未来版本的重中之重。

**3。该产品有哪些隐私问题？**

我们假设电子邮件将包含个人识别信息(PII)，并且这些信息可以包含在预测某人文本的机器学习模型中。因此，如果有人删除了一封电子邮件，我们也会删除在该电子邮件中接受培训的客户模型。

**4。我们与竞争对手相比如何？**

我们最大的竞争对手 GeezMail 已经为他们的电子邮件客户提供了预测文本，所以我们正在迎头赶上。今天他们提供更多的语言。但是，它们不会针对每个客户进行定制，因此我们的预测文本会迅速变得更加准确，并以客户无法做到的方式针对我们每个客户的“声音”进行定制。
