# 如何为求职准备 GitHub:web dev

> 原文：<https://www.reddit.com/r/webdev/comments/90xmpw/how_to_prep_your_github_for_job_seeking/?utm_source=wanqu.co&utm_campaign=Wanqu+Daily&utm_medium=website>

你好，

我目前正在招聘，大多数分享 GitHubs 的人都没有维护好个人资料和项目，所以我决定为积极求职者写一个小指南。我希望它能帮助一些人(包括我),我也希望有反馈和补充来丰富它。

*(英语不是我的第一语言，所以在语法和拼写上要小心)*

如何为求职准备 GitHub

不管正确与否，技术招聘人员往往会从你的 GitHub 个人资料中推断出很多关于你的信息；由于越来越多的招聘人员要求查看或通过简单的谷歌搜索找到它。对于求职的开发者来说，将 GitHub 视为简历的延伸是很重要的。

考虑到这一点，我根据过去几周我采访的人在 GitHub 帐户中看到的“错误”整理了一些要点。

我主要关注开发人员，因为大多数想法可以推广到任何配置文件。

> 我迁移到 GitLab 是因为 M$太烂了

让我们快速解决这个问题。

首先，专业的成年人不会那样说话(在公共场合),所以要确保你把这种观点留给自己，而不是像我在一个固定的知识库的 README.md 中看到的那样。虽然我的公司目前不使用任何微软产品，但我们不会关闭根据技术和财务优势选择一个产品并在未来使用它的大门。我不需要不能以开放的心态评估技术的开发人员。

Gitlab 看起来是一个很好的选择，虽然因为我自己使用 GitHub，并且我正在 GitLab 上查看 20 个 GitHub 配置文件，所以我更熟悉 GitHub 的界面。这并不意味着你必须使用它，下面的大多数建议应该或多或少翻译。(如果你正在使用 bitbucket，显然你从未向我们申请过...)

整理您的个人资料

首先，让我们来看看你应该对你的总体档案做些什么。

添加你的照片、简短的简历和其他相关网站的链接

拥有一份完整的个人资料给人的印象是你关心你的 GitHub，并且你是一个有条理的人。是的，我们有你的简历，可能还有你的 LinkedIn，但这是三者中最差的，会给人留下最持久的印象。

一幅真实的图片胜过一幅卡通、漫画或一个符号。

如果你在找工作，重要的是你看起来干净；避免你通常不会发给雇主的照片，包括裸体、饮酒、忍者、黑客或恐怖分子伪装，或者任何给人感觉你闻起来比看起来更糟糕的照片。

在 GitHub 个人资料上，你最好看起来放松友好，所以避免过于正式的照片，但要保持专业。如果有疑问，可以选择“休闲星期五”的风格。

最后，你的毕业托加画像并不是一张好照片。不要用。这让你看起来没有经验。

使用@ reference 链接招聘人员应该注意到的其他回复。您参与或管理的公司、小组或其他存储库。

包括您个人网站的 URL。如果你有一个，它看起来不像废话，确保它包括在内。(提示:如果它看起来像垃圾，也许你一开始就不应该有这个网站...)

**链接到您的其他相关个人资料。**如果你在 StackOverflow、CodePen、Behance、Dribble 或其他网站上活跃，确保你所有的个人资料都链接在一起。不要链接到你在 furries，weeaboo 和 cosplay 论坛上的个人资料，因为它与你的职业经历无关。

> **关于意外链接的说明**
> 
> 谷歌是一个很好的链接生成器，所以如果你不想让你的招聘人员找到他们，就不要在所有这些不同的网站上重复使用同一个昵称，因为当我们做社交媒体检查时，我们会找到他们。
> 
> 我不知道谷歌了多少次 GitHub 用户名，通过谷歌找到的关联，我发现了他们的副业，从与雇主竞争 Upwork 到经营非法养狗业务，以及在 Instagram 上看起来像是招揽生意的事情。

为招聘人员选择您的固定信息库。

如果你在找工作，出现在你个人资料顶部的固定存储库不应该是你最常访问的存储库的快捷方式，而是展示你最令人印象深刻的存储库的橱窗。也许他们是一样的，但最有可能不是。

作为一名招聘人员，我不在乎看你的 bash_profile，而是钉上你想向招聘人员突出显示的存储库。他们应该是相关的，并显示已经完成的工作，等等..

⚠️锁定你希望招聘人员查看的前三个知识库。

不要固定教程

请意识到，我们可以看到很多简历，所以如果你的回购来自教程或新兵训练营，有可能我已经看过三、三十次了。过于简单的项目或者 80%来自教程而你已经填空的项目不能告诉我你的能力，或者更糟的是，它们告诉我那是你能做的最高水平。拥有它们没有问题，只是不要突出它们。

如果这就是你的全部，最好不要钉任何东西，把那些回购叫做它们本来的样子:教程。不要为了自己的发明而试图通过它们，也不要试图夸大它们；它们是给初学者的教程，不会给任何人留下深刻印象，正如我之前所说，我可能已经看过那个教程了。

将技术演示与相关技术联系起来...

*...在使用过时技术的大型项目之前。*

例如，如果你正在寻找一份 SPA 开发人员的工作，那么就把 Angular/React/Vue 技术演示放在你三年前正在做的 Laravel + jQuery 上面。

如果你和大多数人一样，你并没有运行一个成功的开源项目，你日常生产的大部分代码都是你雇主的财产，所以你可能会因为没有一个令人印象深刻的项目来展示而感到难过。别担心。大多数人不会。

而不是未能留下深刻印象的未完成的和大多被放弃的过于雄心勃勃的项目(谁没有 1 或...10?)在你选择的框架中，关注一些有趣的小型技术演示。如果你已经在专业地工作，那么肯定有一个你解决了的问题会成为一个有趣的演示，你只需要快速重写它或者请求你现在的老板授权发布它。

**如果你是初学者，**我建议你做一些演示，展示你对 CS 理论的理解，即使是 React 或 Vue 中的冒泡排序或树遍历可视化。与使用 Udemy 教程中的 todo 应用程序的同龄人相比，你会看起来像个天才。

*寻找灵感:*

如果你有 CS 之外的爱好(比如音乐),那就加入进来。不知道，做个[小合成器](https://github.com/JanCVanB/vue-synth/blob/master/src/components/Synth.vue)什么的。

移除无用的叉子

我收到了很多提交的资料，这些资料包含了数百个零贡献的分叉库。首先，它让人很难看出你实际上在做什么，其次，它给人的印象是你不明白 Git(和 GitHub)是如何工作的，你在用 forks 作为书签。

有些人可能会用叉子来“填充”他们的个人资料，错误地认为这会给招聘人员或他们的朋友留下深刻印象。这并不令人印象深刻，它让你看起来像一个装模作样的白痴。不要这样做。

⚠️叉库只有当你有贡献，否则使用星或跟随。

清理您标有星号的存储库

这会让很多人生气，但不管怎样，你的标有星号的知识库向访问者表明了你的兴趣，它们应该随着你技能的成熟而改变，所以你标有星号的内容会影响你的观感。

⚠️，你应该检查一下你最后几页的星星，把可能会被误解的那一页删掉。

初级 Vs 高级

粗略地说，寻找高级职位的人应该关注包含复杂项目的知识库，避免太多初学者或教程回购(除非你正在贡献)。

想象一下，查看一位高级 react 开发人员的个人资料，发现她感兴趣的最后两个回复是[Ili akan/JavaScript-tutorial-en](https://github.com/iliakan/javascript-tutorial-en)和 [kay-is/react-from-zero](https://github.com/kay-is/react-from-zero) 你可能会怀疑候选人是否真的拥有她声称拥有的技能，或者她是否仍处于教程级别。

另一方面，如果你显然是在申请一个初级职位，或者表明你目前正在学习该技术，那么在你的故事中，在这两个知识库中打上星号“有意义”。

白帽子 Vs 黑帽子

我喜欢安全方面的东西，从白帽的角度来看，我认为这对于任何关心它的开发人员来说都是一件好事，但是不要让一长串晦涩难懂的破解工具出现，除非你申请的是安全职位。我看过一个人的简介，他有一长串 wifi 破解和网络钓鱼工具，然后是一堆共享枚举器，最后是几个密码暴力工具。就讲故事而言，这不是最好的一个。

整理你的展示库

所以一旦你选择了几个仓库来展示；确保它们是可访问的。以下是他们应该具备的最低要求:

所有项目都应该有一个有趣的自述文件

出于对你所关心的事情的热爱，请给你的项目添加一个描述。这听起来是显而易见的，但我无法告诉你有多少次我看到没有自述文件的 repos，或者使用任何骨架工具创建的默认自述文件。

以下是您在自述文件中应具备的最低要求:

描述

你需要让有 60 秒时间的人很容易理解这到底是怎么回事，以及为什么这能让你成为一名优秀的程序员。

请确保您至少回答了以下问题:

*   它是做什么的？写一个短句，列出工作功能。

*   **什么事？清楚地指出代码应该产生什么。是 web、桌面、手机 app 还是图书馆？**

*   **使用了哪些技术？**列出所有对这个项目有贡献的重要框架和库。对于不一定熟悉地球上每一个框架的招聘人员来说，知道这是 Laravel 和 Vue 还是 React 和 Expressjs 是很有用的。

*   项目的目标是什么？您只是在试驾一项技术，还是这项技术已经或即将在某个地方投入使用？

*   **项目处于什么阶段？**清楚地表明你与它的关系。它是完整的还是正在进行的工作？如果是一项正在进行的工作，指出什么已经完成，什么还在等待。

*   是否有一些已知的问题或事情没有做好？如果是，请列出它们，因为当/如果我发现被强调的缺点时，我会比我自己发现它们更宽容。

告诉我看什么

如果你坚持展示一个大项目，可能会有很多绝对无趣的样板或“管道”,所以不要犹豫，指出哪里是最有趣的部分。

如果这是你参与的一个分支，明确你一直在做什么，如果很难确定，那就不要展示这个项目。

如何运行它

你绝对应该对如何运行它(或者如果它是一个库的话，如何使用它)有一个清晰而精确的解释。

运行项目的演示版本必须有一行活动，如`npm run`、`graddle serve`、`docker run`..或者你的框架所使用的任何东西。

在这个时代，没有什么理由要有一长串手动依赖项和运行任何东西的预设置。

**这是你应该争取的:**

```
# serve with hot reload at localhost:8080
npm run dev

# build for production with minification
npm run build

# build for production and view the bundle analyzer report
npm run build --report

# regenerate Element component styles in theme/ from element-variables.css
npm run theme 
```

演示

如果你有一个可访问的现场演示，请直接在描述中链接。如果您无法进行现场演示，请使用[屏幕录像机](https://www.producthunt.com/alternatives/giphy-capture)；如果这是过分的要求；至少要有几张截图。

https://asciinema.org/的控制台录音似乎很酷，但我没有试过。
