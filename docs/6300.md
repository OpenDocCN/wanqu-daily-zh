# 买而不建:为了乐趣和利润而避免运营

> 原文：<https://jrott.com/posts/why-buy/?utm_source=wanqu.co&utm_campaign=Wanqu+Daily&utm_medium=website>

维护和管理服务或构建定制服务是工程师的普遍愿望。这通常是一个重大错误，最终会耗费大量的时间和金钱。构建定制版本的愿望似乎来自几个地方:

1.  希望建造比购买便宜。
2.  他们公司的流程很特别，所以行业标准的东西行不通。
3.  他们需要完全控制服务的内容。
4.  为了避免供应商锁定

所有这四件事都没有你想象的那么真实和重要。当某样东西对你的业务至关重要或能提供显著的竞争优势时，它就值得建立。否则，使用您的云提供商或其他 saas 的服务可能是值得的。经营自己的东西有很大的运营负担和巨大的机会成本。如果你只能从中得到一件事，那就顺其自然吧:构建东西很有趣，但在凌晨两点被传呼谈论一个处理客户分析的 Rube Goldberg 系统装置就不有趣了。

## 运行服务并不容易[](#p-aligncenter-running-services-isnt-easy-p)

保持系统正常运行需要时间和精力。建造它们并不是花费最多的地方。相反，这伴随着复杂系统的运行和维护。大多数企业系统需要一个工程团队来保持它们的运行。雇佣工程师并不便宜，而且为了保持大量团队的协调，还引入了额外的复杂性。这些都会导致决策速度变慢。

更慢的决策发生是因为需要更多的团队来维护更多的服务。这些团队然后需要一起工作和协调。突然之间，为了做出改变，有一百万个团队需要被通知，并且必须管理移交。这可能导致管理者的领地和更多的政治，因为现在有更多的团队和更复杂的组织结构。

如果您遵循 DevOps 模型，构建服务的团队也将最终维护它。团队需要维护的移动部分越多，他们开发新功能的时间就越少。这是痛苦的，尤其是在产品快速发展的年轻公司中。放慢寻找产品市场契合度的时间，以此换取经营自己的东西，是一种糟糕的交易。

您还必须考虑组织中存在的卓越运营水平。说白了，就正常运行时间而言，你更信任谁——亚马逊还是你？对于你赖以生存的服务，答案可能是你自己。不过，其他系统可能会因为获得的时间和关注减少而遭受损失，因为维持它们正常运行的成本变得越来越难以衡量。

## 厂商锁定[](#p-aligncenter-vendor-lock-in-p)

在这一点上，你可能会想“但是我不想被一个供应商的特殊系统所束缚”。我对此的反驳是，内部系统也存在锁定。最常见的版本是电子表格的保管者。现在，如果你要问“什么电子表格？”这是一个公平的问题。但是对于一些重要的内部过程来说，它已经变成了电子表格的保管者。大多数大公司至少有一个这样的电子表格。如果你在一家大公司工作，你可能会意识到这是一个保守的估计——可能有很多。

电子表格的保管者会捍卫他们的流程，并不希望不惜一切代价去改变它，因为他们担心如果流程自动化或不再必要，他们会被解雇。你也可以在工程团队中看到这一点，他们成为数据库或票务系统的管理员。突然之间，你有了一个糟糕的系统，没有人想要主张摆脱它，因为他们的同事确信，如果发生这种情况，他们会丢掉工作。当粗心的人试图修正这一过程时，这也为他们制造了一个政治陷阱。

作为票务系统的管理员通常也不是那么有趣。这是陷入无聊工作的好方法。这也意味着您最终得到的系统对业务来说不是最重要的，而不是让外部公司来接手。外部公司可能会专门解决这个问题，并因此积累了更深层次的专业知识。当然，除非你有点邪恶，想找个可怕的项目把人们流放到那里。

所有这些都使得被局限于某个供应商并不像大多数人想象的那样令人担忧。无论你做什么都有锁定。你要避免的事情是给任何供应商批发定价权。这可以通过确保企业的关键竞争优势在内部来避免。

## 工程时间贵[](#p-aligncenter-engineering-time-is-expensive-p)

雇佣软件和系统工程师并不便宜。作为一个群体，我们也倾向于低估我们的时间。想想你经常听到“哦，我可以在一周内完成”，或者“这很容易”。幸运的是，这只是 Reddit 或 hacker news 上的一条评论，但如果它在工作中，那么它通常会变成一场苦战。

通常用总拥有成本(TCO)来表示拥有或维护服务的成本。这通常很难计算，因为许多计入总拥有成本的因素没有被跟踪。你会遇到的主要问题是，这不仅仅是工程师的成本。我们关心的指标是工程师可以生产的其他东西的机会成本。

构建定制的另一个原因是独特的公司流程。通常有这样的想法，你不能定制软件使其工作，或者它的成本比仅仅构建它要高。虽然这些都是合理的建造理由，但这种情况比你想象的要少。许多流程在大量业务中是共享的。此外，随着时间的推移，流程往往会变得臃肿。与业务流程匹配所需的大量定制工作是无法完成的。举个荒谬的例子，这发生在我工作的一家公司:

1.  我们推荐文章的过程很复杂，需要大量的模糊数据连接
2.  我们可以建立自己的数据库系统，专门处理这种情况。
3.  几个月的紧张发展过去了。
4.  事实证明操作这个东西是很难的，我们不知道为什么一些查询会关闭系统，为什么我们的客户会抱怨这些建议。

你真的不希望那是你。为甚至不能正常工作的东西构建和维护一个产品是令人沮丧的。当事情变得如此复杂，以至于现有的工具都无法胜任时，您应该问是否所有的复杂性都是领域的基础，或者您使用的模型是否有缺陷。

## 失去焦点[](#p-aligncenter-loss-of-focus-p)

运行您自己版本的服务所带来的一个重要问题是，这是工程师必须注意的另一件事。重要的事情是有限的。接下来发生在你正在运行的所有非核心服务上的事情通常是某种形式的疏忽，它们被保持在勉强够好的状态。

随之而来的问题是，从事这些服务的每个人通常都试图摆脱它们。毕竟，没有人想做老板不关心的事情。所以你最终会有大量的机动动作，因为人们正试图换队，这增加了阻力。

使这个问题更加复杂的是，不产生收入的东西经常被忽略。是的，你的 CI/CD 系统绝对重要，但高管们很容易忽略这一点。这导致了一种失败模式，在这种模式下，您拥有一个内部工具的失落花园。然而，如果你付钱给某人做同样的事情，这是他们的事，所以他们继续工作。

## 机会成本[](#p-aligncenter-opportunity-cost-p)

在许多方面，构建服务的最大问题是机会成本。原因不是工资，而是还能做些什么。这基本上和你在公司开发一次性功能来完成销售时看到的问题是一样的。最大的不同是，工程是在对他们自己做，所以可能会对造成的损害故意视而不见。

工程师喜欢建造东西，许多人喜欢控制所有的按钮和旋钮。很多时候这是好事。毕竟，任何超级酷的东西都是这样建造的。当这种冲动存在时，如果没有对所做决策的商业影响的敏锐感觉，问题就出现了。

你应该问的问题是，除了调整你自己的东西或建立一个新的内部系统，还能做些什么。答案通常是花更多的时间来提出正确的架构，或者开发实际的面向客户的功能，而不是去救火。

拥有大的操作足迹通常会降低速度，减少每个工程师发生的变化。想想大公司和创业公司的速度差别。这不是因为创业公司雇佣了更聪明的人，而是因为大公司的任何变化都会带来大量的东西。

这是一个工程不能只考虑正在构建的软件的领域。相反，你必须考虑整个产品的健康。这是关于建立对客户有用的东西，放弃不重要的东西。通过密切关注核心项目，事情会做得更快。产品附带的工艺和维护工作也较少。

## 总结[](#p-aligncenter-summing-up-p)

这些原因可能都不适用于你的情况。有很多很好的理由去建造。但是，如果你还没有考虑过是否可以只是买一些东西来解决一个问题，而不是自己去构建，那你就应该考虑了。

想想凌晨 2 点被传呼是否值得。如果你愿意被传呼，也要考虑是否有人能比你更好地维护它。你比我更了解你的生意和产品的细节。然而，这是一个你应该问的问题，而不是挥挥手就走开。

我建议你优先考虑购买而不是建造，除非建造能为企业提供真正可持续的优势。随着时间的推移，减少长期的操作负担是保持开发速度和快乐的最简单的方法之一，同时也减少了组织的成本。
