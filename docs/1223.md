每年六月，[苹果](https://martech.org/big-news-apples-event-today-tv-macbook-watches-aplenty/)和[谷歌](https://martech.org/google-io-keynote/)都会举行发布会，宣布他们的最新技术。在每一次宣布中，两家公司似乎都在重申他们更大的公司目标和价值观——苹果坚持其设计和销售优质产品的承诺，而谷歌则加强其收集和组织世界数据的使命，最终目的是通过广告将结果货币化。

这些目标在该公司的许多业务决策中再次出现，最近是通过两家公司推广的不同应用程序索引框架。长期以来，深度应用程序内容一直被锁定在应用程序代码中，很大程度上没有索引，搜索引擎爬虫无法访问。爬虫能获得的唯一内容是来自各种应用商店和公司网站的应用标题和描述。

现在，苹果和谷歌都宣布了在应用程序中索引深层内容的方法，这种情况已经发生了巨大而不可逆转的变化。对于那些对最前沿的数字营销策略感兴趣的人来说，SEO 的概念必须扩展到包括深度应用内容的优化，以纳入苹果和谷歌的索引。

苹果对“产品”的全面关注，使该公司能够从这个主要基于应用的世界中获得最大收益。在苹果的理想场景中，网络只是作为一个无形的层，将应用程序链接在一起，并允许应用程序成为自己唯一控制的私人和公共内容的显示层。

相反，谷歌将从基于网络的世界中获得最大收益，在这个世界中，数据可以最容易地收集、组织和分发，因此谷歌可以成为互联网的表示层。虽然应用程序是其数据收集策略的一部分，但谷歌已经在[项目](https://www.google.com/webdesigner/)上投入了大量资金，这些项目允许开发者创建 HTML5 网络内容来与应用程序竞争，从而将用户返回到基于网络的生态系统。

本文是关于苹果、谷歌和其他生态系统中应用程序索引的新战略和机遇的三部分系列文章的第一部分。

这第一篇文章关注苹果新的 iOS 9 搜索 API 如何鼓励和激励以应用为中心的数字生态系统，以及这对寻求为苹果用户提供深度应用内容的营销人员意味着什么。这是一个全面的指南，你作为一个搜索引擎优化需要知道苹果公司进军搜索工程，它对你公司的应用战略意味着什么，它如何影响你公司的搜索引擎优化战略，以及如何利用这些新的机会来表面你公司的应用和网站在苹果搜索。

该系列的后续文章将重点关注谷歌的应用索引机会和相关策略，以及随着可穿戴设备和其他非标准设备应用和设备索引的增长，我们将面临的未来应用索引挑战。

## 什么是苹果搜索&为什么它对搜索引擎优化很重要？

如果你想向 iOS 用户展示你的应用内容，苹果搜索可能会成为你搜索引擎优化策略的关键部分。像谷歌一样，苹果搜索依靠一个索引(或者更准确地说，多个索引)来组织可以在搜索结果中排名的一组应用程序屏幕。苹果将他们的索引和与这些索引连接的软件统称为“苹果搜索”

通过在 Spotlight 或 Siri 中执行搜索，或者通过用户在 Safari 的地址栏中键入查询时出现的 Spotlight 建议结果(在点击“enter”之前)，可以显示索引的应用程序屏幕。这意味着用户可以在 Safari 上开始搜索，并最终被导向深层链接*，而不会看到一个谷歌结果*——即使谷歌是“默认”搜索引擎。

为了实现这一点，苹果引入了公共和私人索引的概念，并为 iOS 开发者提供了三种方法来为苹果搜索索引他们的应用程序屏幕:NSUserActivity、CoreSpotlight 和 Web Markup。

尽管仅限于 iOS 应用索引(抱歉 Android 哥们，你的应用不能在这里玩)，但苹果新的搜索和索引框架将是一个重要的流量渠道，尤其是对于有时完全绕过谷歌算法的苹果用户。与谷歌不同，苹果提出了两种索引方法，它们不需要相应的网页来驱动应用程序索引:

### **1。安全性索引**

在这种索引方法中，苹果将“用户活动”编入索引，你可以把它想象成一种应用程序屏幕的“书签+ JsessionID + cookie”快照，通过导航了解用户如何到达那里以及他们过去如何与该屏幕交互。苹果将这些快照称为“NSUserActivities”，当开发人员将搜索资格标记添加到应用程序代码本身时(不需要网站)，用户访问带有标记的屏幕时，苹果可以将它们编入索引。

每个 NSUserActivity 还可以与 contentAttributeSet 相关联，content attribute set 包括相关的元数据，如标题、关键字和描述。这些信息用于创建搜索结果的外观，并帮助确定排名。每个 NSUserActivity 都应该有一个唯一的标识符和/或包含一个指向其相应 web 内容的 URL 链接。

### **2。CoreSpotlight 索引**

CoreSpotlight API 一直允许日历和邮件应用程序等原生 iOS 组件在 Spotlight 搜索结果中包含项目。这一功能现已向应用商店中的所有应用开放。这种索引方法有点类似于在 SEO 中提交 XML 站点地图，但是索引文件是与您的应用程序清单一起提交的，它是一个不同的、特定于 iOS 的代码，在一个名为" CSSearchableIndex 的文件中创建

在这种索引方法中，要被索引的应用程序屏幕被称为“ CSSearchableItem (s)”，并且每个屏幕都与一个名为“唯一标识符的标签相关联每个 CSSearchableItem 可以与一个“CSSerchableItemAttributeSet”相关联，该属性集包括相关的元数据，如标题、关键字和描述。CSSerchableItemAttributeSet用于确定算法相关性并填充搜索结果。

每个CSSearchableItem/unique identifier组合也可以与域名标识符相关联，以帮助在数据库中对某些类型的应用程序屏幕进行分组，以便可以将它们作为一个组添加到 CSSearchableIndex 中或从其中删除。例如，唯一标识符可能与特定的照片屏幕相关联，而域标识符可能指示相册中的所有照片。

需要注意的重要一点是，NSUserActivity 和 CoreSpotlight 的搜索结果元数据*来自应用本身的代码*(而不是来自网站)。这意味着 SEO 需要比以往任何时候都更早地参与到应用程序开发过程中，以确保苹果搜索优化的标题、描述和关键字标记在发布时就包含在应用程序中(就像 1999 年发布新网页一样)。

除了 NSUserActivity 和 CoreSpotlight，苹果还允许你使用网络爬虫和深度链接标记来索引你的深度应用内容，类似于谷歌。深度链接是一个过程，其中使用 web 标记将 web URLs 链接到其对应的应用程序屏幕(称为 URIs)，以便爬虫可以理解 URL 和 URIs 之间的连接。此选项*是否要求应用程序内容具有可抓取的相应网页内容:*

### **3。网页标记索引**

在这种索引方法中，苹果的新网络爬虫“Applebot”从与应用清单一起提交的营销和/或支持 URL 中索引应用内容。Applebot 可以抓取您的网站，并基于以下标记协议索引相应的应用程序屏幕:

*   Twitter 卡片标记(Twitter Card Markup):Twitter 卡片标记可以包含一个协议来引用应用程序屏幕的深层链接。
*   [App Links 协议](https://applinks.org/documentation/) : App Links 是一个外部深度链接标准，也是脸书和必应使用的(也适用于 iOS 和 Android)。
*   [苹果的智能应用横幅](https://developer.apple.com/library/ios/documentation/AppleApplications/Reference/SafariWebContent/PromotingAppswithAppBanners/PromotingAppswithAppBanners.html):苹果创建的一种协议，当从 iOS 设备访问网站时，会在网站上显示一个特殊的苹果横幅。横幅要么提示用户打开应用程序(如果已安装)，要么从 App Store 下载。

此外，尽管它不会帮助你索引新的应用程序屏幕，但 Applebot 可以抓取其他两个协议，寻找元数据以显示在搜索结果中:

*   [Open Graph](https://ogp.me/) :脸书的协议，让网络内容更容易分享。苹果支持完整的协议，如果 OG 标签已经到位，Applebot 可以抓取它。
*   Schema.org:一种在网络上被广泛接受的外部标记标准。Apple Search 正在推出有限的支持，包括以下模式:AggregateRating，Offers，PriceRange，Interaction Count(喜欢，查看，评论)，Organization(电话号码)，Recipes，SearchAction(搜索的登录页)，ImageObject 和 Actions，仅限于:拨打电话号码，获取路线，播放音频或视频文件。

Web 标记对 SEO 来说很棒，因为 SEO 可以像我们习惯的那样，通过向网站添加标记来启用和控制索引和元数据。iOS 应用开发者只需要生成 URI 深度链接，并与 SEO 团队分享。如果应用 URIs 已经到位，这可以是一个非常快速和简单的实现，不需要任何开发资源或应用更新。

**重要提醒:**当网页上的 iOS 深度链接被点击时(无论是从你的网页，甚至是脸书或推特)，它们都会启动一个“ openURL 命令，告诉手机操作系统从浏览器切换到一个应用程序——openURL*必须在你的应用程序代码中启用*才能在应用程序中打开深度链接。

在将深层链接包含在 web 标记中之前，请确保您的应用程序确实可以打开这些链接。如果您完全依赖 Web 标记方法进行应用程序索引，您仍需要对您的应用程序进行此更改并提交更新。这是一个小细节，但很重要。

## 苹果如何在苹果搜索中对应用进行排名？

鉴于全球对隐私问题越来越敏感(想想欧盟的“[被遗忘权](https://searchengineland.com/library/legal/legal-right-to-be-forgotten)”)，苹果已经开始关注隐私问题，将其作为一个关键的差异化因素，以及他们产品和服务的卖点。苹果新的应用索引框架展示了通过引入多种应用索引来保护用户个人信息的独特努力:

*   私人“设备索引”，这是一个只有特定用户 ID 才能访问的个人索引，就像每个苹果用户的一个小型个性化数据库
*   一个公共的“云索引”，其中包含可从任何苹果设备通过 Safari 中的 Spotlight、Siri 或 Spotlight Suggest 访问的内容

由于这种架构，开发者可以(也应该)允许用户的私人应用程序屏幕保存到私人设备索引中。任何应用程序中的私人屏幕，如直接消息和用户仪表盘，理论上现在都是可索引的。

**注:**谷歌也使用类似的个性化搜索历史，但从未公开描述为单独的“索引”或开放给开发者访问。谷歌几年前就测试了“桌面搜索”服务，最近还在尝试将 Gmail 的个人电子邮件结果编入索引。我们预计谷歌和苹果的这一趋势将会持续并扩大到未来。

像谷歌一样，苹果也没有透露他们搜索算法的确切细节，但他们描述了一些应该考虑的关键排名因素。苹果的许多搜索排名因素侧重于确定内容在私有设备索引和公共云索引中的排名。SEO 需要被衡量和负责任地对排名产生积极影响，因为苹果已经说过，“恶意或考虑不周的[应用索引策略] *实现*”将被惩罚“*或完全压制*”(把那些黑帽子收起来)。

更全面的苹果搜索排名因素列表如下:

### **正排名因素**

*   **App 安装状态。**设备上安装了应用程序吗？(已安装的应用程序似乎更受青睐)
*   **个性化应用参与。**个人是否参与应用程序中的屏幕？这是基于苹果从会话分析中确定的结果所花费的时间。
*   **App 结果点击率。**用户是否经常点击搜索结果，而不是选择另一个结果或再次搜索？
*   **关键词/标题。**应用程序标记中的“关键字”和“标题”名称中的关键字是否与用户的查询匹配？
*   **聚合参与度。**许多*用户如何使用应用程序屏幕？*
*   Web 上的结构化数据。结构化数据是否正确实现？
*   **规范的应用 id。**同一个屏幕是否通过多种索引方法(NSUserActivity、CoreSpotlight 和 Web Markup)与一个唯一的 ID 或 URL 相关联？
*   Web URL 的强度/受欢迎程度。与深度链接相关的网站有多受欢迎？(想必这是基于 Applebot 的抓取。)

### **负面排名因素**

*   **低参与度。**很少用户使用应用程序屏幕吗？(由会话分析确定的参与度)
*   **过度索引。**app 在指数中是否有很多低参与度或无参与度的屏幕？
*   **退货。**用户在查看应用程序后会立即返回搜索结果吗？
*   **关键词滥发。**开发者是否在关键词字段里塞进了太多不相关的关键词？
*   **填隙物。**是有东西覆盖了 app 里的内容还是阻止用户访问？
*   Javascript(仅网络版)。Javascript 是否阻止了 Applebot 抓取你的站点来寻找新的 app 深度链接？
*   **低星级，低评论量，差评。**苹果公司并没有明确地将这些负面排名因素称为苹果搜索，但它们是 App Store 的负面排名因素，因此我们期待苹果公司在这里以类似的方式对待它们。

苹果建议采用多种索引方法来优化应用程序的可见性，但重叠的方法将不可避免地在各种索引之间产生重复。例如，私有内容可以索引 NSUserActivity 和 CSSearchableItem，公共内容可以索引 NSUserActivity 和 Web 标记深层链接。

对于控制 Applebot 的效率来说，这显然*不*理想，所以苹果*强烈*建议将每个 NSUserActivity、CSSearchableItem 和 Web 标记深度链接关联到同一个唯一标识符和/或 URL。这是 Applebot 的典范版本，它对苹果如此重要，以至于他们甚至把它作为苹果搜索算法中的一个排名因素。

## 我如何决定哪个索引选项对我的公司最有战略意义？

iOS 开发者可以指出他们认为他们的应用程序内容应该包含在哪些索引中。并不是所有的 app 内容都必须被索引，所以你的公司需要回答的第一个问题是:哪些内容应该被纳入苹果索引，哪些不应该？

然后，因为不是所有的索引方法都可以将内容分配给公共和私有索引，所以要问:在应该被索引的内容中，哪些内容应该是公共可访问的，哪些内容应该是私有的？这些问题的答案将是你的应用索引计划的开始。

生成所有应用程序屏幕及其相关活动的列表，然后绘制出内容的公共或私人性质，这是记录这一过程的一种有组织的方式。一旦确定了应用程序内容的私有或公共性质，就可以选择适当的索引方法来满足这些需求。nsureactivities 和/或 CoreSpotlight 索引方法均可用于索引专用设备索引的内容，而 nsureactivities 和/或 Web 标记均可用于索引公共云索引的内容。

习惯了 Google 有些直接和开放的索引标准的 SEO 可能很难理解将应用程序内容添加到 Apple 搜索索引的 3 种不同方式以及这些方法是如何交互的。

NSUserActivity 是唯一一种能够私有地*和公开地*索引内容的索引方法。NSUserActivity 会话分析的参与度统计是排名算法的一部分，因此苹果建议首先关注 NSUserActivity 索引方法。

开发人员可以指出是否应该对某个活动进行私有或公共索引。然而，苹果不会自动公开索引那些被标记为公开索引的活动。苹果仍然首先将 NSUserActivity 索引到私有设备索引，然后一旦一定数量的人访问了它，就将其“提升”到公共云索引。

这一新框架最独特的方面之一是何时以及如何将特定的非安全性活动提升到云指数中。苹果的代表没有进入细节，所以这方面的框架可能有点不清楚，直到推出。幸运的是，他分享了一个图表，似乎表明一个应用程序中的每个屏幕都可以有各种各样的活动，有些是私人可索引的，有些是公共可索引的。开发人员必须将每个可索引的 NSUserActivity 与资格代码相关联:“ var eligibleForSearch ”(私有索引)和“var eligibleForSearch”+“var eligibleForPublicIndexing”(公共索引)。

在大多数情况下，像用户仪表板屏幕这样的东西会有一个可公开索引的默认 nsurectivity——对于以前从未访问过该屏幕的任何人来说，这是一个理想的公开搜索结果。代表屏幕非默认状态的特定于用户的 NSUserActivities 只应该是私有可索引的，因为它们只适合于触发它们的特定用户。

由于默认和定制的 NSUserActivities 都与同一个应用屏幕相关联，因此访问该屏幕(公共和私人)的任何人都会生成参与度数据，以有利于该屏幕的可公开索引的默认版本。因此，默认公共版本的屏幕是提升到公共云指数的更强候选。此外，由于参与度指标是一个积极的排名因素，无论用户与应用程序的历史交互情况如何，默认的公共版本的屏幕也可能在所有搜索中排名更高。

## 如何将不同的苹果应用程序索引协议结合起来使用，以使 SEO 受益？

开发者可以单独使用 NSUserActivity 索引方法，但是除了 NSUserActivity 索引方法之外还使用 Web 标记索引方法有一些*显著的*优势(这是 Apple 不能单独从 NSUserActivity 推断出来的):

1.  当 Web 标记就绪时，深度链接的应用程序屏幕会自动添加到公共云索引中。(不，要等到一定数量的用户发现了你的内容，你才能跨过苹果的“促销门槛”)
2.  当 Applebot 在您的网站上找到合适的 Web 标记时，这些网页也可以添加到 Apple 的索引中。如果没有安装应用程序，网页可能会在 Spotlight Search 中排名，而不是在应用程序中排名。(**注:**这意味着 web 标记索引方法给了你两次排名的机会:app 和 Web。这是一个巨大的竞争优势，因为*苹果只索引已经提交到 iTunes connect 的网页或者包含这个应用深度链接标记的网页*。苹果机器人不像谷歌机器人那样是一个漫游爬虫。)
3.  “网站人气”有助于提高你的应用在苹果搜索中的“相关性分数”，这是苹果搜索算法的一个核心方面。
4.  来自网站的结构化数据是苹果搜索算法中一个独立的排名因素。
5.  [可以从你的网站引入动作模式](https://schema.org/docs/actions.html)，为你的应用程序的搜索结果添加额外的视觉元素(#pizzaz)，使其对用户更具吸引力，并提高搜索结果的点击率，这是苹果搜索排名的另一个因素。
6.  额外收获:这也让你的深层链接在谷歌搜索中被索引了一半(这将在本系列的下一篇文章中详细讨论)。

深度链接和用户活动——无论是私人的还是公共的——只有当应用程序安装在用户的设备上时(或过去)，才会显示为结果。当用户的设备上没有安装应用程序时，苹果将显示应用程序商店结果或索引网页内容(由苹果机器人抓取网站的网页标记发现)。

当用户获得网站结果时，苹果指望网络开发者实现智能应用横幅，将用户从网站引至应用商店的应用下载页面。苹果的智能应用横幅广告非常有助于推动从传统网站推荐接近网站的网络流量的下载；但如果苹果搜索的最终目标是引导人们远离网站，进入一个只有应用的数字空间，这种策略就错失了良机。

注意:与谷歌相比，这显然是一种不太直接的体验。没有安装应用程序的用户点击谷歌搜索结果中的谷歌深度链接，会立即被发送到 Google Play 或 iOS 应用商店的应用程序下载页面。

苹果之前已经测试过将这些点击直接发送到 App Store，但是收到了想要网站选项的用户的反对。虽然新的解决方案满足了这些用户，但它可能无法将尽可能多的搜索转化为应用下载。如果用户不觉得必须点击智能应用横幅，或者(更有可能的是)如果 web 开发人员根本没有在他们的网站上启用智能应用横幅，这就特别站不住脚了。

## 结束语

SEO 从业者必须不断发展，以跟上“SEO”定义的扩展随着新的 iOS 9 应用程序索引的发布，SEOs 也必须发展，以适应苹果对搜索工程的新理解。

苹果现在提供深度应用索引，这与谷歌提供的应用索引机会明显不同。这在很大程度上是由于它的公共和私人索引框架，以及营销人员可以用来将他们的深度应用程序内容编入索引的方法。

苹果挖掘 iOS 用户参与度指标的能力使他们能够将参与度作为排名算法的核心部分，而不是像标题和描述这样的传统排名因素。与苹果目前在 iOS 应用商店中处理搜索的方式相比，这是一个显著的进步，后者严重依赖标题和关键词来确定应用排名。

越来越多的机会瞄准更多具有深度应用内容的关键词，这应该会激励应用开发者在应用开发的早期阶段稍微更多地关注营销，并迫使 SEO 和应用开发者之间建立更具战略性的持续合作伙伴关系。

在所有关于新的苹果搜索和应用程序索引的讨论中，一些人认为尚未得到充分解决的大问题是关于搜索量和跟踪。尚不清楚新的苹果搜索工具将获得多少总搜索量和参与度，特别是与谷歌搜索相比(以及谷歌在其搜索结果中显示 iOS 应用深度链接的新能力)。

用户会被应用程序的深层链接而不是传统的网页内容所吸引吗，或者他们会知道其中的区别吗？用户会开始倾向于使用 Spotlight 搜索来查找新的应用和应用内容吗？他们还是会更喜欢应用商店搜索，还是苹果搜索工具之间的界限会继续模糊，导致单一的以应用为中心的 Spotlight-Safari-Siri 仪表盘界面，可以快速跟踪应用对网络的完全控制？

对于一些应用程序缺乏网站对等性的公司和其他应用程序具有大量私人或个性化内容的公司来说，苹果搜索目前代表着唯一的机会让他们的应用程序内容被索引，但苹果搜索的整体影响仍有待观察。

虽然苹果搜索为苹果设备上的新曝光提供了许多机会，但开发者必须在苹果新的应用索引框架内工作，并使用苹果新的专有方法为各种索引进行优化。成功将受到搜索量的限制，苹果搜索商店可以从其忠诚的观众那里获得多少，而搜索优化计划的影响将更难衡量。

由于苹果对隐私的推动是谷歌无法比拟的(至少，现在还不是，除非彻底改变该公司的数据收集做法)，我们预计苹果将在注重隐私的功能上进行更多关键投资。请记住，苹果的商业模式是基于他们自己的产品盈利，而不是基于帮助营销人员盈利。如果苹果认为用户会为隐私(或隐私的幻觉)支付更多的钱，他们会毫不犹豫地限制来自营销人员的数据。

相反，如果苹果认为他们可以通过出售苹果搜索数据获取更多利润，我们可能会在未来看到针对营销人员的新产品——收费。(还记得那些 IAD 有多贵吗？)苹果在向营销人员提供最基本的分析方面一直比较晚，所以苹果搜索很少甚至没有提供我们在谷歌分析中已经享受到的许多指标。

虽然我们很欣赏苹果公司的视频解释，以及他们给 SEO 和开发者准备这些变化的时间，但是许多关于这项技术的问题仍然没有答案。我们不禁要问:苹果是否计划使用某种版本的 QDF 来确定将安全性提升到公共云指数中？

换句话说，参与可公开索引的非安全活动(如 NHL 应用程序中的布鲁因斯游戏)的人数突然增加，会导致该活动在总浏览量更多但最近流量较少的活动(如布鲁因斯队名册屏幕)之前被提升到公共云索引吗？更新鲜的活动会因为时效性排名靠前吗？相反，当人们停止参与新的用户活动时，他们会被“降级”回到私人设备指数吗(或者，比方说，被更近的布鲁因斯游戏取代)？
