# 硅谷招聘不是精英管理——艾琳·勒纳的博客

> 原文：<http://blog.alinelerner.com/silicon-valley-hiring-is-not-a-meritocracy/?utm_source=wanqu.co&utm_campaign=Wanqu+Daily&utm_medium=website>

注:这篇文章改编自我在 Quora 上写的一篇[回答。](http://qr.ae/ISwaz)

编辑:我正在做一些事情，希望能让事情变得更好。查看[面试。](http://www.interviewing.io)

在过去的三年里，我一直在雇佣一些人。起初，我是一名工程师，然后是一名内部招聘人员，现在是我自己的技术招聘公司的老板。直到最近，我还十分确信，创业世界是一个精英社会。

我错了。大多数招聘经理口头上说他们在寻找强大的 CS 基础、激情和伟大的项目，而不是血统，但实际上，这个系统崩溃了。令人悲伤的事实是，如果你在纸面上看起来不怎么样，而你申请的是一家拥有强大品牌的初创公司，除非你在公司里认识人，否则你获得面试机会的几率非常小。

人们关注技术面试本身，认为这是招聘过程中不公平的一个潜在来源，但我要关注的是我认为更大的一个问题:刚刚迈出你的第一步。

系统如何崩溃

假设你是一家新创业公司的技术联合创始人，这家公司已经有了一些发展，并且有一堆积压的工作要做。幸运的是，你已经筹集了足够的钱来雇佣一些工程师。很有可能，你是唯一的技术人员，所以你可能是全职从事工程招聘的人。这对你来说有点糟糕，因为这可能不是你认为自己最擅长的，而且由于机会成本，这对公司来说也有点糟糕。然而，你可能做得很好，因为 1)你真的很擅长销售公司的愿景，因为你对它如此着迷，2)你有技术才能和直觉，根据你通过经验获得的一些内部启发来评估人。因为这些启发，你可能最终会让那些看起来聪明的人进来，即使他们在纸面上看起来不怎么样。

最终，事情进展顺利，你的初创公司获得了更多的资金，你决定要回去做你认为真正的工作。你雇佣一名内部招聘人员全职处理你的招聘渠道。

这是事情开始变糟的地方。因为招聘人员一般来说不是技术型的，他们不得不依靠快速识别的属性作为能力的代理，而不是依靠一些内部的能力指标。 *好的*，你想，*我要给我的招聘人员一些关于什么样的好候选人的指导方针*。你甚至可以创建一些招聘规范来帮助他们。这些招聘规范，无论是一份正式文件还是一份标准清单，都倾向于关注候选人的属性，最大限度地提高候选人优秀的几率，同时最大限度地减少得出这些结论所需的专业知识。这些属性的示例包括:

*   顶尖学校的计算机科学学位
*   在顶级公司工作过
*   特定语言/框架的知识 <sup>1</sup>
*   一些年的经验

这个系统工作…某种程度上。如果你的公司有一个非常强大的品牌，你可以承受很高的假阴性发生率，因为总会有一个候选人的旋转门。虽然这些标准并不伟大，但它们显然表现得足够好，足以延续它们的存在。

不幸的是，这个系统创造了一个巨大的伟大工程师的长尾，即使面对明显的工程劳动力短缺，他们也被忽视了。

副业呢？

民间智慧表明，如果你在纸面上看起来不太好，拥有一个很好的兼职项目组合可以帮助你获得成功。我希望这是真的。然而，**除非你的副业引人注目，容易让外行人理解，或者是用你申请的公司的 API 构建的，否则它很可能会被忽视**。

问题是兼职项目种类繁多，对于非技术招聘人员来说难以区分。对于这样一个人来说，带有一点个人色彩的复制粘贴式 UI 教程和一个从头开始的功能完整的 JavaScript 框架是一回事。对于一个有技术背景的人来说，区分这些类型的项目有些耗时，而对于一个以前从未编写过代码的人来说，这几乎是不可能的。因此，虽然出色的兼职项目是能力的一个巨大指标，但如果阅读简历的人不能(要么因为缺乏特定领域的知识，要么因为时间的考虑)区分出色和平庸，信号就会消失在噪音中。

澄清一下，我并不是不鼓励建筑材料。用自己的时间构建东西是一种很好的学习方式，因为你会遇到各种不确定的挑战和困境，而这些是你在其他情况下不会遇到的。很少有什么能让你更好地准备编码面试，测试你是否知道 web 如何工作，是否有不错的产品意识，是否能设计 DB 模式，等等。只是有了产品并不能保证你就能进入这个领域。

对此我们能做些什么？

哀叹非技术人员首先筛选候选人是愚蠢的，因为这不会改变。然而，可以改变的是他们如何进行过滤。我能想到一些办法来解决这个问题。

*   找出哪些属性是成功的预测因素，而不仅仅是像血统这样唾手可得的东西。这很难。[我试着去做](https://blog.alinelerner.com/lessons-from-a-years-worth-of-hiring-data/ "Lessons from a year’s worth of hiring data")，而[谷歌做到了](http://www.nytimes.com/2013/06/20/business/in-head-hunting-big-data-may-not-be-such-a-big-deal.html?_r=0 "In Head-Hunting, Big Data May Not Be Such a Big Deal")。我希望更多的公司做这种事情，并公布他们的结果。
*   寻找其他评估候选人的方法，这些方法至少和简历过滤一样便宜有效，但实际上是更主要的能力指标。
*   建立一个免费/低成本的精英 CS 课程，任何人都可以参加，但大多数人都无法完成，因为课程的难度和严格的评分。将此打造成为一个足以让企业信任的强大品牌。通过这种方式，优点可以有效地与血统联系起来。这也真的很难。不过，我知道很多公司都在攻击这个领域，我很兴奋地看到未来几年招聘将会发生怎样的变化。

我的第二个建议是我将在这里谈论的，因为它是最容易实现的，并且它的一些部分已经就位。

为了避免只根据血统来判断，而不是传统的简历/求职信，在纸上看起来不怎么样的申请人可以通过 1)编码挑战和 2)写作样本来申请特定的公司。

理想情况下，编码挑战可以快速自动评分。这也应该是一个有趣的问题，让人们对公司的工程文化和他们正在解决的问题有所了解，而不是泛泛而谈。如果编码挑战太教科书化了，某个聪明人的子集不会想在上面浪费时间。自动化编码评估的工具已经存在(例如 [InterviewStreet](https://www.interviewstreet.com/ "InterviewStreet") 、[codi ty](http://codility.com/ "Codility")和 [Hackermeter](http://www.hackermeter.com/ "Hackermeter") )，但是它们目前还没有以正确的方式融入到应用程序流程中。在 InterviewStreet 和 Codility 的情况下，候选人必须已经在某家公司的管道中，才能应对编码挑战，这意味着任何在纸上看起来糟糕的人都将在开始编写一行代码之前被淘汰。就 Hackermeter 而言，候选人首先编写代码，然后希望与一家公司匹配。这种方法有很大的潜力，但除非 Hackermeter 能够将自己确立为权威的启动应用程序，否则候选人将始终受到 Hackermeter 的客户列表的限制。理想的解决方案可能是一个更分散的 Hackermeter 版本，公司可以从他们的工作页面进行管理。

写作样本的想法来自我在进行的[研究。在我测试的许多属性中，有三个具有统计学意义的是语法错误，候选人是否在顶级公司工作，以及你能否从他们的简历中看出他们在之前的每个职位上都做了什么。换句话说，三个属性中的两个与候选人的书面沟通技巧有关。](https://blog.alinelerner.com/lessons-from-a-years-worth-of-hiring-data/ "Lessons from a year’s worth of hiring data")

对于写作样本，我在想象一些不是求职信的东西——人们倾向于把这些写得很公式化，不谈论任何太私人或有趣的东西。**更确切地说，它应该是对你最近正在做的、你很想谈论的事情的简明描述，就像向非技术观众解释的那样。**我认为非技术观众方面至关重要，因为如果你能为非技术观众分解复杂的概念，你可能是一个很好的沟通者，并真正理解你在做什么。此外，招聘人员实际上可以阅读这一描述，并对写作是否良好以及他们是否理解该人的工作做出有价值的判断。如果招聘人员能够做出真正的判断，而不是充当人类关键词匹配者，他们可能会在工作中做得更好。根据我的经验，许多招聘人员都非常聪明，但是他们的技能并没有被充分利用。

一个成功完成的编码挑战和一个优秀的写作样本的组合应该足够强大，足以让某人打电话来进行一轮现场编码。

让钻石原石浮出水面的问题很难，但我认为这些相对较小、现实的步骤可以大大有助于找到它们，并让 eng 招聘进入我们都希望它成为的精英制度。
