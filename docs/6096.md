# 想法很容易，但分类却很难

## 为什么我们应该要求人们提交问题，而不是想法

作为一名项目经理，我一直在努力解决的一个问题是不断有新的想法出现在我面前。以及如何让人们觉得他们的想法是有价值的，即使我们只能对其中的一部分采取行动。现实是，想法不是创新的难点，执行也不是(这句话通常跟在这句话后面)。决定哪些想法，以什么样的形式，以什么样的顺序是困难的。我们项目经理的角色——“想法分类者”的角色非常重要，却被低估了，而且很难做到。我开始相信让人们提交他们的想法实际上是一个坏主意。相反，我们应该要求他们提交问题和假设。

# 创新定义

让我们倒退一分钟，谈谈“创新”的定义。几年前，当我为我的项目管理团队组织一次关于这个主题的演讲时，我注意到我从来没有真正思考过这个词。而不是想当然地认为我知道那是什么。这是我的发现。

> 对于被认为是创新的东西，它需要是新的、有价值的、被实现的和被采用的。

我觉得不用我细说新的或者有价值的，那是大家都“知道”的部分。但是让我们花点时间谈谈另外两个。

## 执行

我想很多人都忘记了一件事，那就是要被认为是“创新”的东西必须是实际完成的。这就是它成为“创新”而不是“想法”的原因。我认为这意味着“创新的想法”不是一个有效的词对。“潜在的创新想法”会更准确。当然，要求实现也意味着它必须是可行的(人类心灵运输是一个想法，但还不是一个创新)。处于目前技术可行的边缘可能使它具有独特的创新性。但事实未必如此。这也可能只是让一个更容易、更优雅的解决方案变得更难解决。

## 被收养的

这一点有些争议，尤其是在软件产品的背景下。当然，有些产品失败仅仅是因为缺乏意识或非常强大的现任者，因此没有被采用。但我认为这是相当罕见的。创新无处不在，不仅仅是产品。如果你发明了一个很酷的新方法来跟踪 sprint burn-down，但是你的团队没有采用它，这真的是一个创新吗？

要进行收养:

1.  你需要解决用户现有的或潜在的需求，或者(在极少数情况下)创造一个新的需求。
2.  目标用户需要知道解决方案(对于一个特性，是可被发现的)。
3.  必须是那些必须改变他们当前行为的用户认为足够重要的需求来采用你的解决方案。
4.  它应该满足需求，而不会给用户带来意想不到的痛苦，或者至少不会带来“太多”的痛苦)。
5.  如果它与解决相同需求的其他东西竞争，它需要是满足该需求的最佳解决方案(无论“最佳”对用户意味着什么)。

产品和功能在所有这些点上都失败了(FWIW，以我的经验来看，大多数都失败在第 1 或第 3 点)。无论他们在哪里失败，这些有争议的想法看起来仍然很像“好”想法。尤其是那些把它们提交给你的人。

希望你能看到实现这个创新的定义比简单地拥有一个你认为有价值的新想法要难得多。因此，为了获得有规律的创新成果，你应该在决定做什么(对“想法分类”)时应用多严格的标准。

## 创意来源

当一个想法出现在你的办公桌上时，它通常只是作为一个想法出现。但是这个想法是怎么产生的呢？让我们先来看看“理想”的方式:

1.  一个**洞察力**产生于数据或研究或者来自[直觉](/intuition-demystified-4ee47343a5c2)。
2.  一个关于洞察力的**假设**被开发出来。或者换一种说法，形成一个清晰的问题陈述。
3.  假设/问题陈述经过**验证**(有需要吗？)并决心符合我们的产品策略。
4.  我们**在当前产品或产品线的背景下头脑风暴**多个不同的**解决方案**，并选择一个来实施。我们找到“最好的”、最可行的解决方案(想法)。

现在让我们看看其他一些“非理想”的方式…

1.  **来自人自身的经历**。这可能与用户的体验有关，也可能无关。“我想要这个功能，所以让我们建立它！”
2.  **来自技术**。IMHO，现在的一堆 AR 产品就是很好的例子。“我们可以在我们面前的桌子上玩这个游戏。为什么？因为我们有技术！”
3.  **从开发者的角度**。“我们建立了这个插件系统，所以让我们添加更多的插件！”
4.  **来自现有特征**。“我们在上一个版本中添加了这个功能，我们来改进一下吧！”

我并不认为其他方式是“错误的”。这里面很可能有很好的想法，应该会被实施。但是要弄清楚这一点，你需要后退并再次前进。也就是说，回到“理想”过程的第 2 步。确定这个想法所要解决的假设/问题，然后验证这个假设并集体讨论解决方案。最初的想法只是对(有希望的)有效假设的一个可能的解决方案。

对于想法的提出者来说，这感觉像是浪费时间，对他们来说，这个想法显然是值得实现的。但是，如果你真的经历了这些步骤，开始对话的想法实现的几率相当小。因此，这个想法分类者进退两难。没人说这份工作容易。

## 问题和假设

那么，如果我们不要求人们提交他们的想法，而是要求他们提交他们的问题陈述或假设，会怎么样呢？如果他们想包括一个可能的解决方案，他们也可以这样做。但是他们不能在不描述问题或背后的假设的情况下提交一个想法。这会如何改变与项目经理谈话的基调？这如何让人们对我们所交付的东西保持一致？这将如何增加我们交付的实际创新的数量？精神食粮。
