# 人数目标、特色工厂，以及何时雇佣那些神话般的 10 倍员工

<time datetime="2019-02-21T00:00:00Z" class="post-date">2019-02-21</time>

当我开始为“更好的 T1”建立一个技术团队时，我做出了一个非常明智的决定，支付高端薪酬来获得人才。我认为这更有意义:雇佣他们会多花一点钱，但产出通常会超过补偿。许多首席技术官同行，有些人走了另一条路。这对我来说是个谜，直到我明白了一切。

## 什么是输出？

在我们开始之前，让我澄清一下我所说的“产出”或“生产率”是什么意思。我不是指一个工程师只是在键盘上以光速敲打航运代码。当我谈论它时，我指的是一系列的事情，比如帮助你的同事，引入新的框架，改进过程，等等。我曾经写过关于这个 T1 的文章。

当然，你不能真的测量它。但是所有的经理都会尝试，当他们把爱丽丝的工资定为 110，000 美元，鲍勃的工资定为 115，000 美元。所以在某种程度上，经理们当然相信他们对每个工程师的相对价值有一些精确的想法。

## 人数目标

让我们剖析一个经典的管理目标:人数目标。在一个典型的工程招聘流程中，首席技术官(或高层人士)大致算出他们需要完成的工作量与他们拥有的工程师数量之比，然后去找首席财务官讨价还价，然后得到这些人的人数和工资范围。然后，这些信息会在整个组织中递归分配，每个招聘经理都会得到一个招聘人数的目标。

假设首席技术官坚决认为他们需要在一年内将工程团队扩大两倍。这涉及到一个初级工程经理。如果你正在进行一个分散的面试过程，那么你就知道制造了一个巨大的代理问题，初级经理被告知他们在公司的成功部分是由他们达到招聘目标的程度来衡量的。当然，他们会降低雇佣标准！

想不到会发生这种事？我已经看过了。我看到招聘门槛开始下降，因为好心人要求更多的资源。以及随着时间的推移，工程人才的平均水平如何缓慢下降。

## 解决错位问题

解决这个问题的正确方法部分是将面试过程和决策*集中化*。任何团队都不应该强加自己的招聘标准，因为有了激进的人员目标，团队中的每个人都会被激励去降低标准。

但让我们也在更基本的层面上:*为什么要制定目标？这就产生了一个潜在的假设，即每个工程师都有大致相同的生产力。实际上，工程师生产力可能非常分散。*

那么，为什么不以某个产出水平为目标呢？当然是因为工程师不会贴上标签说这个是 2.3 倍的工程师，价格 14 万美元，另一个是 4.5 倍的，价格 18 万美元。你不知道！让我们先来谈谈这种关系，因为我认为理解它很重要。

## 成本是生产力的函数

可能令人惊讶的是，成本作为生产率的函数似乎是一个次线性函数。一个 3 或 4 倍的工程师可能要多花 2 倍的成本。这显然不是一条符合直线的物理定律，但我认为大多数认真招聘过的人都会承认，它遵循的不是线性规律。

例如，让我们说一个$k$x 工程师的成本是$k^{0.6}$.因此，对于 2 倍的工程师，我们多付 1.5 倍，对于 10 倍的工程师，我们多付 4 倍。指数的选择在这里有点随意，但关键是要反映出成本比例*小于线性比例*。任何小于 1 的指数都适用于这个论点，请注意，大于 1 的指数在有效市场中是不存在的。没有人会以 2.1 倍的成本雇佣 2 个工程师，他们只会雇佣 2 个 1 倍的工程师。

![kx engineer](../Images/30f3a6526f1b8b9e8324749205fb433a.png)

这似乎是显而易见的。为什么没有人愿意多花一大笔钱来雇佣最高级的工程师呢？让我们把人数目标扔出窗外，代之以*总产量目标*也许我们甚至应该有一个*总工资美元目标*，而不是人数？除了让你的首席财务官相信这一点的挑战之外，这可能还会进一步误导激励。

人数目标通常带有你事先同意的工资级别。如果你仔细想想，这是另一个奇怪的限制——如果更贵的工程师有更高的投资回报率，那么为什么要限制成本(从而限制生产力)？

这些是我一直难以理解的事情。事实证明，您可以形式化一个简单的模型，在这个模型中，即使总成本更高，雇佣两个 1x 工程师而不是 2x 工程师也是合理的。

## 特征工厂和任务开销

对于雇佣“廉价”的工程人才，有一个普遍的观点，那就是大量的任务是直接的、不性感的或者无聊的。也许一个初级工程师不介意整天调整 WordPress 主题，但是一个高级工程师需要更多的挑战。在这个范围的极端，有一类公司经常被嘲笑为*功能工厂*，我怀疑人们会想象一个超级没有经验的工程师的血汗工厂，基本上是更新 HTML 格式的表格或添加跟踪像素。

我很不相信这个论点。一个资深人士会找机会自动化和减少重复的部分，为自己买单。

然而，这个想法有一个小小的变化，我认为它确实证明了雇佣经验较少的工程师是合理的，这与任务开销有关。让我们考虑一个玩具模型:

假设我们有两个工程师，一个叫 Norm 普通工程师，一个叫 tw anda 2x 工程师。假设他们都在一家公司工作，Norm 花了 50%的时间*实际上*在工作，剩下的时间作为“任务开销”浪费掉了。可能是一些簿记工作(去吉拉，创建 Github 拉请求，等待 CI 等等)。这是*每个任务*都要发生的开销。

![2x engineer](../Images/92220185354f8c334e711bc63be0b489.png)

与 Norm 相比，Twanda 的生产率提高了多少？2x？不要！Twanda 创造了 4/3 的价值！一般来说，如果 1x 工程师花费$ c $的时间在“任务开销”项目上，那么$ k $x 工程师将有$ 1 / (c/k + 1-c) $的产出系数。

请注意，花时间参加会议不会产生同样的影响。在一个假设的公司里，90%的时间都花在会议上，一个速度快 2 倍的工程师仍然可以多完成 2 倍的工作(10%的时间不花在会议上)。我的模型说的是*任务相关开销*。

你可以在这个玩具模型中看到，在高任务开销的环境中，一个高产出的工程师的许多生产力收益将会减少。如果你能最大限度地减少任务开销，你真的会从更有效率的人那里获益更多！

## 高产工程师的成本效益分析

现在我们有了一堆假设，让我们可以计算一个$k$x 工程师的单位成本的*产出。我们知道产出因素$ 1/(c/k + 1-c) $和成本$ k^{0.6} $所以每成本*的*产出为:*

$ \frac{1}{(c/k + 1-c)k^{0.6}} $

对于任何给定的$ c $值，我们都可以求出$ k $的最佳值！对$ k $求导，设为零。因为我是一个懒惰的人，我只是[把它插入到 Wolfram Alpha](https://www.wolframalpha.com/input/?i=solve+D%5B1%2F((c%2Fx%2B1-c)*x%5E0.6),+x%5D) 中，作为$ c $的函数的$ k $的最佳值变成了

$ k = \frac{2}{3}\frac{c}{1-c} $

让我们画出$ k $相对于$ c $的最佳值。我不得不把它画在对数刻度上，让形状看起来很漂亮:

![kx engineer](../Images/b398a67ecba691d17687784377f99057.png)

漂亮！让我们从图表上选取几个点来解开这个谜团:

*   极端情况:如果开销是 100%,那么最好的性价比就是雇佣 0x 工程师。
*   如果开销约为 80%,那么最物有所值的做法是雇佣 0.2 名工程师。
*   如果开销大约是 40%,那么最物有所值的就是雇佣 1 名工程师。
*   如果管理费用大约是 20%,那么最物有所值的就是雇佣 3 倍的工程师。
*   如果管理费用大约是 7%,那么最物有所值的就是雇佣 10 倍的工程师。
*   极端情况:如果开销是 0%,那么最物有所值的就是雇佣 x 个工程师。

所以这一切都是为了降低工作的开销。

## 让您的技术团队发挥最大价值

我们谈论了很多工程师在生产率和成本方面的差异，以及如何充分利用他们的价值。好消息是，归结起来只有两件事！

1.  有一个统一的高标准的集中招聘流程
2.  将任务开销降至最低

如果你没有这些东西，就没有必要去雇佣超级高级人才:尤其是你可能会比雇佣普通工程师更好。 [Xavier Amatriain 写了一篇博客文章](https://medium.com/@xamat/cultural-overfitting-and-underfitting-or-why-the-netflix-culture-wont-work-in-your-company-af2a62e41288)得出了类似的结论:不要指望你可以挑选网飞文化的元素并将其放入你的创业公司。你可能必须从你的发展过程和招聘过程开始！

如果你在我写这篇博文之前问我，为什么有些公司为工程师支付高薪，而有些公司没有，我可能会说，有些公司非常专注于技术，因此他们可以真正从真正昂贵的工程师那里获得价值，而有些公司是使用一些现成框架的脚本集合，一个昂贵的工程师不会产生巨大的差异。

我仍然认为这是正确的，但我认为确切的因果关系与这篇文章中假设的模型有更大的关系。举个例子，谷歌(以高薪闻名)有各种挑战，工程师可以长时间独立工作。这降低了(分摊的)任务开销，这意味着他们从一个昂贵的(但更有生产力的)工程师那里获得了更多的价值。其他公司有大量的小项目(因此有很大的任务开销),这意味着他们不应该在市场的顶部支付。

事后看来，这一切对我来说绝对是“显而易见”的，也许你也有同感。至少你知道有一些数学来支持它！
