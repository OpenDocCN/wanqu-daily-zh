它始于有人在公司聊天中发布 AWS 新闻稿。标题是“AWS 宣布 FooBar”。

和往常一样，该公告回避了技术细节，但包含了与该公司产品重叠的令人担忧的大量术语。有人用鬼脸表情来回应😬。其他人用各种方式回答 AWS FooBar 完全不像我们的产品，不管怎样，我们的产品更好，而且，而且…这一连串的回答暴露了他们真实的集体情绪:我们完蛋了。

他们担心是有道理的。

![Cartoon about the unpleasant future.](../Images/35c592943438e454e0a0f608f7759763.png)

在过去的六年里，作为企业软件初创公司的营销顾问，我已经帮助公司在这种情况下导航并取得成功至少八次。在这篇文章中，我解释了为什么每个软件初创公司都应该为这种情况做好准备，为什么最初的恐慌是合理的，以及如何针对“大云”(AWS、Azure、GCP)的替代产品定位产品。

## 每个软件初创公司都应该准备好与 AWS、Azure 和 GCP 竞争

即使您现在没有与大云竞争，也很有可能很快会与大云竞争。

如果您的产品在无服务器、流处理、机器学习、集装箱化、物联网、数据仓库或批处理领域，这一点尤其正确。根据 Rightscale 的一项调查，这些是增长最快的云服务，大型云公司都知道这一点。

大型云公司的大部分收入来自存储、计算和流媒体的计量计费。在将这些基础架构工作负载转化为商品后，大型云公司正在竞相提供更高端的差异化产品，以吸引更多客户使用他们的基础架构。

这在他们产品组合的庞大规模和产品发布的速度中显而易见:AWS 在 2018 年有 182 款产品，发布了 83 款主要产品(发布或主要发布)。<sup>【2】</sup>

## 当 Big Cloud 推出类似产品时，你应该感到担心

在得知一个大的云入侵时，有一种倾向，要我们埋下头。内部邮件将包括解释为什么大云产品“没有真正的竞争力”，“不如我们的好”，以及“只能占据一小部分市场。”

也许这些事情会成真，但至少有充分的理由认真对待这场竞争:

### 1.资源不平衡

大型云公司可以靠蛮力取胜:在工程、营销和销售方面投入大量资源，直到它们跑得比竞争对手更快或更持久。仅仅有一点点更好的产品，或者在市场上有一个小小的领先是不够的。

### 2.复杂的关系

参与大云合作伙伴计划的初创公司，如 [AWS 合作伙伴网络](https://aws.amazon.com/partners/)，将发现自己与竞争对手共享敏感信息和宝贵资源。对他们来说，离开这个项目意味着放弃一个潜在的重要获取渠道。

与此同时，大云不会因为失去创业伙伴而退缩，因此没有动力谨慎对待它获得的信息。

### 3.广泛的范围和影响

AWS 可以登上舞台影响一千人，或者发一封邮件影响十万人，或者播放一个电视广告影响数百万人。利用其广泛的影响力和品牌认知度，大云可以影响人们对市场的感知和决策。

### 4.受制而走不开的听众

比可接触到的观众更好的是被俘虏的观众。大型云公司拥有数百万用户，已经在他们的平台上运行，熟悉他们的产品。只需点击几下鼠标，他们就可以接触到这些受众，从而追加销售和交叉销售产品。

### 5.定价

大多数大型云产品的目标不是盈利，而是将客户带到平台上并增加基础架构的使用(计算、存储、流)。因此，这些产品可以作为亏本产品:定价非常低或没有成本，前提是它们能够推动基础设施使用量的增加。对于创业公司来说，往好里说，这消除了价格竞争的选择，往坏里说，这迫使他们降低价格，却无法弥补。

### 6.容易接近

已经在运行 AWS 的公司的工程师可以在咖啡冷却之前购买、部署和集成 AWS 产品。计量计费意味着没有前期成本或谈判。在同一个平台上意味着工具和服务之间的无缝集成。

### 7.早期决策

过去，决策者必须选择是自行构建解决方案还是从供应商处购买软件。如今，大型云产品的低价、易用性和智力共享为决策者创造了一个新的选择:使用他们的云平台提供的东西。

以前是“购买 vs 构建”现在是“购买 vs 构建 vs 大云”

对于软件初创公司来说，这意味着他们可以在购买过程中更早地被排除在外，在任何对话、评估或功能比较发生之前。

## 如何战胜大云

### 准备

参加贸易展，寻找那些可能会向你透露即将到来的竞争的内部人士，并保持密切关注。提前警告可以让你挖一条更宽的壕沟，准备好应对措施，避免新闻稿发布时全公司范围的恐慌。

拥有一个与大云竞争的先发制人的战略，将有可能利用他们宣布后的第一波炒作，利用他们的活动将注意力吸引到你自己的产品上，并从其他会措手不及的初创公司中脱颖而出。

通过正确的战略和执行，Big Cloud 推出的竞争产品可能会成为初创公司及其市场的积极转折点。

即使您现在没有与大云竞争，当那一刻到来时，良好的准备将提高反应时间和成功的机会。如果你的产品甚至鼓励更多地使用云计算、存储或流媒体，那么你应该做好准备。

### 支持多云

根据 Rightscale 的同一个[调查，84%的企业组织的应用程序和工作流分散在内部、私有云和公共云环境中。](https://info.flexerasoftware.com/SLO-WP-State-of-the-Cloud-2019)

如果你是“AWS 上针对 *X* 的第一解决方案”，而 AWS 推出了针对 *X* 的解决方案，你就成了遥遥无期的第二选择。

如果您的产品可以在各种环境下工作，那么对于想要解决整个组织问题的购买者来说，它将仍然是一个可行的——甚至可能是更好的——选择。这些购买者将需要一个适用于他们整个基础设施的解决方案，而不仅仅是 AWS 上的部分。

对于我从决策者那里听到的购买软件而不是使用云产品的最常见的原因之一就是对多云和混合环境的支持。

与大型云产品相比，多云和混合环境支持可能是一个决定性的优势。而且，与功能上的小领先不同，这种优势将持续一段时间:大型云公司没有什么动力为其他平台制造产品。

有了像 [Kubernetes](https://kubernetes.io/) 和 [Gravity](https://gravitational.com/gravity/) 这样的选项，给产品增加这种能力可能不需要巨大的工程努力。

### 开源前三思

尽管开源产品有助于提高知名度和可信度，但它整体或部分降低了竞争对手的准入门槛。对于任何新的和足够受欢迎的开源产品 *X* ，对于大云来说，引入一个“托管的 *X* ”、[就像 AWS 对 Elasticsearch](https://aws.amazon.com/elasticsearch-service/) 所做的那样，或者将其与他们的安全产品配对并打包为“ *X* for Enterprise、[就像 AWS 对 MongoDB](https://aws.amazon.com/documentdb/) 所做的那样以及[就像 Google 对 Kubernetes](https://cloud.google.com/kubernetes-engine/) 所做的那样，都是微不足道的。

(Kubernetes 的故事有一些有趣的转折:首先，Docker 公司发布了一款开源的容器化产品。当他们推出付费容器管理服务时，谷歌已经开发并开源了自己的容器管理解决方案 Kubernetes，并赢得了市场。然后，在 Docker 发布了托管容器的企业解决方案之后，Google 又一次击败了 Docker，推出了 Google Kubernetes 引擎。)

例如，当 AWS 推出开源软件 Elasticsearch 的“全面管理、可扩展和安全”包装时，它导致该软件的制造商[承认](https://www.sec.gov/Archives/edgar/data/1707753/000119312518266861/d588632ds1.htm):

“…亚马逊与我们竞争潜在客户，虽然亚马逊不能提供我们的专有软件，但亚马逊产品的定价可能会限制我们调整产品价格的能力。”

[开放核心商业模式](https://gravitational.com/blog/open-core-vs-saas-business-model/)的可疑好处，加上它对来自大型云公司的竞争开放的脆弱性——[并不以利用](https://www.cnbc.com/2018/11/30/aws-is-competing-with-its-customers.html)为耻——这就是为什么我建议公司保护他们的产品，不要开源，或者限制他们的脆弱性，如果他们已经开源的话，[就像当 AWS](https://www.geekwire.com/2018/concerned-cloud-providers-confluent-becomes-latest-open-source-company-set-new-restrictions-usage/) [对 Kafka](https://www.cbronline.com/analysis/aws-managed-kafka) 采取相同策略时，Confluent 通过改变他们的许可所做的那样。

### 针对类别而非产品进行定位

现在，公司在对产品进行并行比较之前，必须在“构建还是购买还是大云”之间做出选择，创业公司必须将他们的产品定位为整个“大云”解决方案类别的更好替代或补充。

不要说谁的特性更好。首先，鉴于大型云公司可获得的工程资源，任何功能优势都将是短暂的。第二，在购买过程中，购买者会在很晚的时候进行功能比较，根据他们对这些产品的感知而不是功能来排除大多数选项。

那么，尽管(可能)部署成本和时间更高，为什么你的产品比大云更好呢？

以下是我在采访买家时发现的常见答案，可能适用于也可能不适用于你的产品:

#### 适用于任何地方和任何东西

“我们的产品可以在任何基础设施上部署、运行和集成应用程序。运行应用混合或多云环境的组织可以不受限制地使用该产品，并且无需整合到一个云提供商。”

#### 非技术用户的自助服务

“大型云提供商的产品通常服务于最接近基础设施工作的用户，如工程师、运营商、架构师和开发人员。我们的产品让[…]等非技术用户能够完成他们的工作，而不会受到技术团队的阻碍或增加他们的负担。”

#### 端到端解决方案

“我们的产品是一个端到端的解决方案，不需要工程工作来将零件和其他系统粘合在一起。大云产品更像是构建模块，可以与平台的其他部分很好地协作，但它们本身并不那么有用，甚至不可用。”

#### 专门建造的

“我们的产品旨在解决您所在行业的独特需求，并将继续发展。它可以与您现有的人员、工作流程、工具和系统无缝集成，例如[…]。”

#### 专家合作伙伴

“除了与您的使用案例相关的文档和示例，我们还为所有客户提供专家支持和咨询，无论其规模大小。”

#### 客户驱动的开发

“我们专注于为[…]制造最好的产品，并根据像您这样的人的意见和新需求不断做出改进。”

* * *

如果你仍然不确定如何解释你相对于大云产品的优势，问问那些选择你而不是 AWS、Azure 或 GCP 的买家，就像我对 gravity 和 Netlify 分别对[和](https://www.gkogan.co/case-studies/gravitational/)[所做的那样。](https://www.gkogan.co/case-studies/netlify/)

在开发出与大云竞争的新定位后，将其转化为新的或更新的信息，使公司与该信息保持一致，并通过您的销售和营销渠道将其推向市场。

## 结论

大云的进入并不总是丧钟，但它是严重的。承认来自大型云公司的威胁、做好相应准备并以正确的战略和紧迫感做出反应的创业公司不仅能生存下来，还能茁壮成长。甚至可能长到足以被一个人获得。
