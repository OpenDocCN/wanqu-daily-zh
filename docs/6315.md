## 说话、打字、思考:软件不是案头工作

2020 年 10 月约 7 分钟阅读

 <summary>**TL；开发人员过度优化了打字的人机工程学，但对思考的人机工程学却做得不够。**</summary> 

前几天我洗了一个很棒的淋浴。

这是上午晚些时候(因为最好的淋浴往往是)，我在思考我如何度过一天的时间。作为一名在家工作的顾问，我经常需要证明我的账单和我的时间是合理的，在这种情况下，我证明花更多的时间在淋浴上是合理的。

像我们大多数人一样，我开始我的职业生涯时的印象是，富有成效的一天是在符合人体工程学的键盘上向 Microsoft Visual Basic 6.0 Professional Edition 键入数百行代码，而不是站在完全热的高压淡水流中度过的。然而，我作为一名开发人员的时间越长，我就越不相信我需要在我的办公桌前交付美国高级工程顾问交付的真正令人震惊的电子表格到网络应用程序的商业价值。

为了让你也能证明把一个上午的大部分时间花在清洁温暖的茧中是有道理的，让我们来分解一下，看看有效软件开发的 5 个物理活动。像所有好的列表一样，这是按照时间和重要性增加的顺序排列的。

## 5.讲话

有些软件开发可能不需要任何交谈就能有效。例如，我知道大声谈论 linux 内核开发被普遍认为是不礼貌的。~/bin 的内容太多了，我们就不说了。

但是我参与的每一个商业项目都需要至少一些交流。当人们太忙或太害羞而不敢说话时，高带宽通信的缺乏会使人们很难梳理出需求和解开解释不清的业务问题。

但更重要的是，缺乏交谈会使建立信任和融洽变得困难——这在任何新关系的早期阶段都是至关重要的。作为社会动物，我们特别擅长在口头上这样做，而不是特别擅长用电子邮件和辛辣的子甜言蜜语这样做。

另一方面，我曾参与过一些项目，在这些项目中，交谈是一种掩饰，没有人知道该做什么。十几个人坐在一个房间里，一言不发地聊了一个小时，我们走出来的时候比进来的时候还笨。

因此，在大多数情况下:交谈是至关重要的，但要适量。

## 4.倾听

老实说，我只是为了对称才加上这个的。我要补充的唯一一点是，我们有两只耳朵和一张嘴，所以要么双耳听觉提供了对抗某些选择压力的进化优势，要么我们应该听两倍于我们说的。

无论你喜欢什么，你都可以接受你的智慧。

*(快速谷歌为什么我们有两只耳朵)*

## 3.写作

当然是写代码啦！但也…阅读材料、评论、内联文档、公关描述、代码审查、git 提交；这都是*核心工作*的一部分。很容易将这种元编写视为真正的“代码”编写之上的开销。但是在这些地方有效的写作是你的代码的力量倍增器。

然而更重要的是，以我的经验来看，最好的交流是书面的。

*   它是异步的，这意味着它可以在每个读者方便的时候使用(例如，在傍晚淋浴后)。
*   它可以很容易地分发，并且在共享时没有保真度损失(相比之下，与约翰谈论莎拉告诉你的事情，史蒂夫在会议上说的话，你们都没有出席)。
*   它创造了一个记录，而不是“等等，我们为什么…？”
*   [写作就是思考](https://alistapart.com/article/writing-is-thinking/)！写作迫使你连贯地组织你的想法(至少对一些人来说是这样)。它揭示了你的理解或计划中的缺点或差距。

正因为如此，我鼓励团队沟通主要是书面的。吉拉、slack、电子邮件、trello、博客帖子等等。即使是一面贴有便利贴的墙的高清照片，有时也是不可或缺的建筑路线图。然而它的出版，详细，深思熟虑的写作是💯。

也许比写作更重要的是…

## 2.阅读

刚刚赞美了写自述文件、提交信息、公关描述等的优点，我显然应该鼓励你去读它们。它被称为 README 是有原因的，不仅仅是因为它是首字母缩写词。然而，如果每次有人问我一个已经在自述文件中回答的问题，我就能得到 1 美元，那么我就能得到 3 美元。💰

这是因为我简单地称之为恶性阅读-写作-循环-反馈循环。当人们不更新评论时，人们会被训练成忽略它，所以人们不会更新它，等等。事实上，如果你知道有人在阅读你的 git 提交，他们的质量会迅速提高。即使你以前从未从你的同事那里读到过连贯的 git commit，让他们详细解释一下`finally fix it`是什么意思也不迟。

但是就像写作一样，阅读的价值远远超出了代码库。

我最近开始了一个项目，涉及一个完全陌生的医疗技术领域(ps 你仍然是我最喜欢的病人`01-004`📊❤️).我发现在项目的这个阶段最有价值的活动是阅读。

我们必须解析一种专门的文件格式，对于这种格式有[和](https://github.com/nsrr/edfize)。但是为什么要将所有有用的上下文埋藏在宝石中呢？[文件规范](https://www.edfplus.info/specs/edf.html)并没有那么长，即使需要多次尝试才能理解它。阅读文件格式规范会更容易理解为什么 gem 需要[load _ digital _ signals _ by _ epoch](https://github.com/nsrr/edfize/blob/93566cdc82b160ef319c51908c1c4a19666e2625/lib/edfize/edf.rb#L243)，这反过来会为您手头的问题提出替代解决方案。

没有从阅读这些资源中获得的洞察力，没有一个*邻近的可能的* <sup id="fnref:1" role="doc-noteref">是可发现的，所以不管你在处理什么，去读读读读…</sup>

*   文档(*阅读写得非常好的 [postgres 手册](https://www.postgresql.org/docs/current/index.html)或 [redis 文档](https://redis.io/documentation)是我最近一次将功夫下载到尼奥大脑中的经历*
*   代码(*被大大低估了——我的宝石档案里没有一颗宝石我没有`bundle open` d 过至少一次*
*   日志文件，错误信息，如何阅读火焰图的教程
*   规范、立法、政策文件、NIST 指南、开放存取期刊的原始论文

…你有了想法，只要找到权威文件，然后把它吸收进你的大脑。即使看起来没有什么是一成不变的，与文本的短暂相遇也会留下持久的印象。这就像顺势疗法，但却是真实的。

所以说/听…写/读…最后… *鼓声*

## 1.思考

当你把它归结起来，这个对我来说是主要的努力，然而它是一种隐藏的努力。

我的编程/编码/开发时间有多少实际上只是花在了*思考*问题上？对领域建模，思考边缘案例，在精神上玩抽象。

当你思考是什么造就了优秀的开发者时，这是显而易见的。我最看重的同事不是准确的打字员，而是思路清晰的人。

然而，这种形象依然存在，即打字就是工作，工作就是打字，富有成效的一天是在你办公桌旁的椅子上度过的。因此，我们有双 4k 显示器、机械键盘、aeron 椅、touchbar、vim 快捷方式，任何优化我们敲击电脑的东西。

但是我们对思考的**人体工程学有多少关注呢？**

当我们将“思考”提升到核心工作时，我们自然会开始专门针对它进行优化。总的来说，我们不需要在任何事情面前才能有效地思考，我经常发现最好不要这样。我最清醒的时候总是在我运动的时候，通常是在我锻炼的时候。此外，我几乎可以在任何地方用手机阅读，最好的对话往往是在散步时进行的。

因此，虽然我为我的工作空间的所有人体工程学感到高兴，但我越来越多地发现，编写代码是简单的一部分，在这里我可以收获我从交谈、倾听、阅读和思考中播种的所有精神作物。

为了将这一点提炼为更有头韵的东西，我有时将它描述为软件开发的 3ts…

## 谈打字思维

**说**听；口头讨论。大多数时候，我们需要少量但至关重要的高带宽同步通信。

**打字**代码注释:READMEs，代码评审，PR 描述；以及所有异步通信:项目更新、技术概述、包含后续步骤的电子邮件；这些都是工作中必不可少的一部分，而不仅仅是辅助性的或无用功。也在某些时候输入实际的代码。但是我发现你花在输入其他东西上的时间越多，你花在(重新)输入代码上的时间就越少。

**思考**:(包括，为了头韵，阅读)

说话、打字、思考:这就是我们所做的工作。就我个人而言，我想给自己空间去做好所有的事情。

总之，我要去洗澡了。🚿
