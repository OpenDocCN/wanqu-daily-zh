# 关于新技术的问题。凯兰·埃利奥特-麦克雷

> 原文：<https://kellanem.com/notes/new-tech?utm_source=wanqu.co&utm_campaign=Wanqu+Daily&utm_medium=website>

鉴于协调和沟通淹没了现代软件开发中的所有其他成本，这是一个迫切需要投资的领域，[尤其是当你的团队扩展时](/notes/on-team-size)。

随着时间的推移，我使用一个由少量众所周知的工具组成的框架来在我们复杂的系统中建立共同的理解。当我们想做一些事情，而不是使用*少数众所周知的工具*(在少数众所周知的模式中)，那就是*离开*。

我有一篇关于技术决策和离职的长文要发表。

与此同时，我想分享一个简短的问题列表，十多年来我一直以各种形式使用这个列表来回答令人恐惧的问题。

可怕的问题是这样的:

> “我们应该使用这种新技术 X，它更快，更好，更优雅，更积极地开发，难道你不致力于 Y 公司的人们在这里学习和成长吗，看，我在周末迅速做出了一个原型，并投入生产，这种技术是不是很神奇，嗯，好吧，去他妈的法西斯极权主义国家，我要离开这里。”

(也许这不是一个问题)

如果你曾经领导过一段时间的技术团队，你就会有这样的对话。(或者，在极少数情况下，错过了进行这种对话的重要机会)

### 这些问题

他们的偏见并不特别微妙。他们不应该在一起。它们也不意味着是一系列要检查的盒子或要跳过的环。

1.  我们试图解决什么问题？(技术永远不应该作为目的而引入)

2.  我们如何用当前的技术解决这个问题？(如果答案是我们不能，那么我们很可能还没有足够深入地思考这个问题)

3.  我们是否清楚新技术带来的新成本？(监控、培训、认知负荷等)

4.  我们当前的堆栈有哪些方面使得以经济高效的方式(在资金、人员或时间方面)解决这个问题变得困难？

5.  如果这种新技术是我们目前所做事情的替代品，我们是否承诺在未来将一切都转移到这种新技术上？或者我们是否在为同一个问题提供多种解决方案？(又名“这个解决方案会杀死并吃掉它所替代的解决方案吗？”)

6.  我们认识并信任谁使用这项技术？我们跟他们谈过了吗？他们对此怎么说？他们不喜欢它的什么？(如果他们不讨厌的话，他们还没有深入使用过)

7.  什么是低风险的开始方式？

8.  你是否召集了一个由资深人士组成的混合学科小组，讨论了上述每一点？哪里有记载？
