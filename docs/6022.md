# Martech 的问题，以及为什么 Martech 实际上是为工程师设计的

由于我在风险投资领域呆过一段时间，并有营销背景，很多人问我关于 martech，或为营销人员建立的技术。这些是好生意吗？他们应该使用/正在使用哪些工具？

总之，我讨厌 martech，认为 martech 作为一个品类会衰落，大部分 martech 业务都不会很成功。我认为这其中有几个原因没有得到很好的理解，但如果你理解了它们，它可以释放一些对企业家来说仍然很大的 martech 机会，并帮助营销人员了解哪些技术值得押注，哪些技术需要引进。主要的误解是，成功的 martech 其实是给工程师的，不是给营销人员的。先说为什么会这样。

**Martech 是对工程约束的回应**
我之前陈述过的一个有争议的观点是，科技公司的营销职能通常是对工程约束的回应。如果你没有足够的工程师来建立一个系统来管理绩效营销的投标，你就雇佣一个营销人员。如果你没有可以从事 SEO 的工程师，你可以雇佣营销人员。如果你不能建立一个伟大的电子邮件系统，你聘请营销人员。大多数关键的营销角色都是手工任务，可以通过工程更好地解决。最聪明的营销人员意识到了这一点，开始通过第三方工具来自动化他们的大量工作，如果可以的话，最好是第一方工具。这就是 martech 在过去十年中的爆炸式增长。实际上，营销人员对公司的责任很重要，如果不是严重不足的话。例如，我负责让新人尝试在 Grubhub 网上订购，并在他们订购后让他们回来。我的团队使用了许多 martech 工具来完成这项工作。

工程约束正在放松
尽管在公司内部雇佣工程师来解决这些问题仍然极具竞争力，但我遇到的每家科技公司的工程约束都在(慢慢)放松。与 Grubhub 在我们公司的类似阶段相比，今天的初创公司和技术公司有更多的工程师从事更多的功能(由于工程技术的改进)。

这些工程约束的放松意味着 martech 公司必须与公司的工程师竞争解决营销问题的最佳方法。除了公司里有更多的工程师来解决这些问题之外，工程师现在更有可能想要解决这些问题，或者拒绝将这些工具作为最佳实践。成长团队已经出现，致力于解决营销团队购买软件解决的许多传统营销问题:电子邮件、搜索引擎优化、登陆页面优化、入职培训等。

Martech 现在发现自己处于一个竞争更加激烈的环境中，因为在“建造或购买”的等式中，“建造”比过去更有可能。此外，如果公司内部的工程师决定构建而不是购买解决方案，很多时候他们构建的解决方案比 martech 提供商提供的解决方案更有效。这并不是说科技公司的工程师比 martech 公司的工程师更优秀；科技公司内部的工程师拥有不公平的优势。工程师不仅可以为他们的公司构建解决方案，直接构建他们公司的需求，而不是采用一些通用的解决方案；他们还可以更容易地集成这些工具所需的数据，以做出正确的决策。例如，众所周知，许多 martech 工具很难集成转换数据，对于终身价值数据来说更是难上加难。使用内部构建的工具更容易做到这一点。

平台也限制了 Martech 的业务范围
Martech 公司也面临着来自整合另一面的压力。通常，martech 公司会整合到一些其他系统中:像谷歌和脸书这样的广告公司，像交易所和需求方平台这样的广告技术公司，电子邮件服务提供商和电子邮件客户端等。事实是，这些 martech 公司在一个平台上构建了增值功能，为客户提供额外的价值。现在发生的事情是，这些平台要么集成了那些最好的功能，所以你不再需要 martech 公司了，要么删除了支持它的访问权限，因为平台实际上不想要那种程度的透明度。

【Martech 能成功在哪里？
因此，这些公司让平台窃取它们的功能或切断让它们成为可能的入口，让客户公司的工程师自己构建更深层次的集成。因此，如果大多数 martech 解决方案在与内部工程解决方案竞争时处于劣势，或者平台开始与之竞争，那么哪种 martech 工具具有优势呢？

**选择 1:利用数据网络效应**
martech 蓬勃发展的一个重要例子是外部数据变得比内部数据更重要。如果一个 martech 工具可以从多家公司收集数据，并从这种聚集中产生一种数据网络效应，从而帮助所有公司以一种它们自己无法做到的方式进行改进，那么它们就非常有说服力。筛选科学就是一个很好的例子。通过被数千家公司用作欺诈提供商，他们拥有任何单个公司在确定交易是否欺诈时都不会拥有的数据。

**选项 2:管理痛苦**
同样，在 martech 中，与一群关键运营商或供应商的集成也是非常合理的。石蕊在历史上是一个经典的例子。众所周知，电子邮件提供商对在他们的系统中呈现什么以及如何呈现有着苛刻的规定，而且他们也不是很透明。工程师和设计师讨厌为电子邮件编码，他们很难记住所有不同类型的电子邮件客户端的所有规则。Litmus 允许你在发送电子邮件之前预览所有主要客户的电子邮件，以发现错误，并且通常成为一个无所不包的电子邮件 QA 工具。内部没有工程师想要构建它，他们永远不会像 Litmus 那样擅长做这件事，因为 Litmus 已经为数十亿封电子邮件做了这件事，所以它看到了更多的案例，并与电子邮件提供商有更好的集成。另一个消除工程难题的例子是 Heap Analytics，它自动标记事件，消除了建立新的分析供应商最痛苦的部分之一。

**选项 3:利用跨端网络效应**
一个更现代的例子是客户数据平台公司细分和细分。这些公司与营销人员用于各种目的的数百家其他公司进行整合:网络分析、绩效营销的转化跟踪、崩溃报告等。整合这些公司可以节省工程师的时间，因为他们只需整合一次，他们需要的任何其他解决方案现在都可以立即启用。这些整合不仅有助于营销，也有助于产品和工程。这些公司在客户和其他技术提供商之间创造了一种交叉网络效应。一旦你进行整合，数据平台公司就很难被淘汰，因为它们在你的所有流程中是如此的整合。

**真正的答案:改变目标客户**
好吧，所有这些都是很好的选择，但他们实际上有一个共同点:他们真的把目标客户转移到了工程师而不是营销人员身上。当然，营销人员可能是要求解决方案的人，但是选择解决方案是因为工程师喜欢它。一个工程师必须做的许多事情是痛苦的，尽管工程师喜欢解决他们自己的问题，但如果你向他们展示价值，他们会感激你的。所以我非常看好伪装成 martech 的 engtech 公司。除了上面的例子之外，这方面的其他例子还有数据可视化平台，如 Mode 和 Periscope。

**额外选择:挑选合适的营销客户**
martech 公司的另一个非常成功的策略是为营销对组织的成功更为重要的公司类型制定有针对性的解决方案。虽然营销在大多数科技公司的重要性正在下降，但它正在蓬勃发展的一个领域是电子商务公司，它们的主要剧本是产品交付的物流，品牌+绩效营销推动所有销售。产品是线下交付的东西，因此产品和工程团队比其他职能部门更服从于营销，而且因为产品是线下交付的，这些团队的工程师通常比其他公司少。Narvar 是电子商务跟踪的一个很好的例子。Buffer 是社交媒体营销的一个很好的例子。Canva 是一个很好的工具，可以帮助设计营销活动和社交媒体帖子的创意。

对于企业家来说，Martech 是一个非常具有挑战性的领域。如果你要解决这个问题，有不同的策略，如数据网络效应，痛苦管理和维护，以及交叉网络效应，使建立一个可持续发展的业务更有可能。接近正确的客户，无论是在角色上(工程)还是在空间上(电子商务)，都会让这条路更容易走。如果你有任何关于建立一个伟大的 martech 企业的其他建议，请随时在评论中留下。
