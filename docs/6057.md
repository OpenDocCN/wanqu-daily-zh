# 不可测量的度量。非理性繁荣

> 原文：<https://lethain.com/metrics-for-the-unmeasurable/?utm_source=wanqu.co&utm_campaign=Wanqu+Daily&utm_medium=website>

随着组织变得越来越复杂，它们通过无形的抽象来管理，这些抽象越来越偏离它们所代表的现实。

在最小的团队中，决策者可能每天都在深入研究代码。再大一点，你说的是短跑中的任务。更多的增长，你在谈论任务的集合，并采用像“史诗”这样的花哨词汇。在一百多名工程师中，你可能主要谈论工作主题，关注几个[关键计划](https://lethain.com/infrastructure-planning/)。在数以千计，你可能会谈论你的资源配置决策的投资框架。

无论路径如何，度量都是您的最终目的地。

度量是非常有效的抽象。一个[格式良好的指标](https://lethain.com/goals-and-baselines/)有足够的上下文来理解你做得有多好，将该表现与历史表现联系起来，并定义一个清晰的、可分级的目标。

带有目标的示例指标可能是:

> 在第四季度，20%的部署被[隐瞒](https://martinfowler.com/bliki/CanaryRelease.html)，比第三季度的 10%有所上升，但没有达到我们 25%的目标。

这是强大的，有几个原因。首先，历史背景描述了我们前进的方向，尤其是当它与一张显示未来的图表结合在一起时。第二，目标是否已经达到是非常清楚的，并且历史向量告知目标有多困难的一般感觉。最后，也是最重要的一点，人们可以在对潜在主题了解最少的情况下进行评估。

领导复杂公司的人很有能力理解底层工作，但他们只能谨慎地打这张牌。有太多的需求需要他们的关注，而度量标准成为了一个伟大的过滤器，用来区分从远处观察和深入观察的区域。

组织需要一些时间来获得新的能力，以满足对度量的贪得无厌的需求。好的度量标准需要对底层数据有深刻的理解，而建立这种熟悉感是真正的工作。度量时代的开始以度量选择的变动为特征。每一次学习都激发了您的度量目录的细化，并且大多数新的度量需要新的仪器、管道和卫生设施。

这需要时间，并且是困难的工作，但这是可预测的工作。你投入更多的时间，学习更多的东西，你的度量选择就变成了一个持久的、有用的现实的物化。

不管怎样，通常都是这样的。随着我在定义和使用度量标准方面越来越有经验，我越来越多的时间花在处理不容易度量的领域上。我有时会被描述为*不可测量的*领域。

当我遇到新的基础架构领导者时，我的第一个问题通常是:

*   “您如何衡量贵公司的安全性？”
*   "你如何衡量你的团队的生产力？"

答案各不相同，但总的来说，两者的衡量标准都有许多不足之处。这两者面临的挑战非常相似，因此我将特别研究安全性的度量。

## 测量安全性

当我们谈论测量安全性时，很容易确定描述您想要的结果的*输出指标*。然而，通常情况下，安全指标的分子或分母都是不可知的。例如，您可能希望设定一个检测入侵的目标，但是您肯定不知道分母是什么，因为这个数字包括未检测到的入侵。

同样，在您确实知道分子的情况下——例如，已知安全漏洞的数量——通常情况下，唯一可接受的目标是零。这些力量共同作用，使“零度量”激增，例如设置“零违规”、“零内部行为者事件”、“零受损系统”等目标。

“零度量”是二进制度量的一种特殊形式，唯一有用的数据是你是成功了(达到零)还是失败了(超过零)。这些不是特别有用，因为它们提供了非常粗略的性能评级，实际上只是通过/失败。更糟糕的是，它们是不可诉的。假设这不是一个被忽视的领域，如果你有零个漏洞，你可能会像有四个漏洞一样优先考虑下半年的相同工作。

如果我们从这些二进制指标退一步，有许多非常有用的指标值得考虑来衡量我们的安全性，例如:

*   带[CVE](https://cve.mitre.org/)的库升级有多快？
*   您最早的活动服务器有多长时间了？
*   超过 mTLS 的请求百分比是多少？
*   有多少人可以通过 SSH 访问服务器？
*   有多少人对服务器有根权限？

挑战在于，这些都是*输入指标*，它们对于衡量计划的进展非常优秀，但是对于衡量计划的质量却很弱。当然，我们加密了 100%的服务器存储，但这在多大程度上提高了我们的安全性？

依赖输入指标的另一个问题是，它们太多太具体，很难综合成有用的理解。这意味着他们未能发挥重要作用，为高层领导提供一个抽象概念，让他们能够有效地理解某个领域的绩效。

## 综合指标

将这些挑战放在一起，我们发现了一个有用的定义，即什么是组织绩效的良好衡量标准:

1.  它应该暴露足够的层次来检测性能随时间的变化，
2.  不同层次的变化应该指导有益的行动，
3.  它们应该可以在有限的背景下进行评估，并且
4.  它们应该足够宽泛以抽象出重要的复杂性。

在这一点上，我不认为有一个*自然*安全度量具有这些属性，但幸运的是，我们不必依赖自然度量。我最近的方法是创建*合成度量*，将输入度量打包成有用的抽象。

一个很好的例子是，如果您设想创建一个“服务器安全等级”，根据其输入指标为每台服务器分配一个从“A”到“F”的字母等级。这里的细节有点正交，但也许一台安全等级为“A”的服务器使用不到 7 天，运行最新的操作系统，并且具有非常有限的 root 访问权限。每天你都要评估每台服务器的安全等级，然后你就可以开始跟踪不同等级服务器的分布情况。

不要详细讨论每个输入指标，相反，你可以开始围绕改变分数分布来设定目标。例如，您的目标指标可能是:

> 80%的服务器的安全等级为 A，高于第 4 季度的 60%，达到了我们 75%的目标。

这用非常简洁的东西代替了大量的复杂性。你仍然可以围绕每个字母等级的定义进行高杠杆的讨论，但是人们可以在不深究这些定义的情况下了解情况。您可以测量分布变化来评估长期进展。如果等级分布向下移动，您可以深入了解哪个输入指标导致了这种移动，并使用它来确定纠正措施。

总的来说，我认为这种方法非常有用，但是它不是免费的。这种方法的主要缺点是设计、检测和发展综合指标的复杂性。根据您使用的工具，维护这种度量的开销可能难以支持，但是我认为随着事情变得更加复杂，这是一种值得添加到您的方法中的强大技术。

人们是如何为不可测量的事物创造衡量标准的？

顺便说一句，出于与测量安全性相同的原因，测量生产率是具有挑战性的。我以前写过一些关于这方面的文章，既总结了 [Accelerate 对生产力](https://lethain.com/accelerate-developer-productivity/)的定义，它侧重于使用行业研究来选择*正确的*输入指标，也在我的[系统思维简介](https://lethain.com/systems-thinking/)中试图将输入指标总结成一个可理解的模型。

我的想法在这里有所发展，关注内部开发人员调查(但不是 CSAT 分数，而是参与率和主要关注点变化的速度)和围绕工作流可用性的综合指标(基于广泛的输入指标，如构建、测试和部署速度和可靠性)。
