# 分享我们的工程阶梯——RTR 着装规范

> 原文：<http://dresscode.renttherunway.com/blog/ladder?utm_source=wanqu.co&utm_campaign=Wanqu+Daily&utm_medium=website>

去年，我在纽约首席技术官峰会上做了一次演讲，主题是如何以及为什么要将结构引入工程组织(此处提供幻灯片)。尽管有潜在的缺点，我还是喜欢结构和它能给组织带来的清晰。我不会在这篇文章中过多地谈论为什么，但我认为是时候我应该帮助做些什么了。

创建工程阶梯(即工程组织的工作描述和级别)是一项艰巨的任务。如果你做一件半心半意的工作，你可能会引起比你解决的更多的问题。我向我的团队展示的第一个梯子是基于 Foursquare*的梯子。这非常简单，每个工程“属性”都有一到两个描述性元素(技术技能、沟通技能等)，而且据说对他们的团队非常有效。我希望这种提供更少细节的风格会让人们不那么痴迷于梯子，不太可能试图在“技术细节”上谈判他们的水平。男孩，我错了。几乎立刻就有工程师和我争论他们的水平是否合适，整个组织充满了焦虑和困惑。我留下了慷慨解释的空间，这让我很难解释我真正想要和期望的是什么。

除了阶梯在我的团队内部造成问题之外，我们还很难在面试中评估候选人，并决定将他们聘用到哪个级别。尤其是在更高层，成功的标准到底是什么样子还不清楚。因此，我们与我的技术主管和工程经理一起，重新编写了阶梯，使其更加具体。这对审查和晋升委员会以及招聘过程都很有帮助。

从那以后，我私下与全国各地寻找灵感的几位工程经理分享了我的梯子，今天我们为了大家的利益将它公之于众。我分享这个梯子并不是说它是工程梯子的全部，而是希望给你一个比你在其他地方可能看到的更彻底的起点。毕竟，没有人真的是从零开始写这些梯子，无论我们是直接从其他公司(就像这家公司在 points 所做的那样)，还是间接从我们过去的经验中，所以数据点越多越好。

事不宜迟，我在[电子表格](https://docs.google.com/spreadsheets/d/1k4sO6pyCl_YYnf0PAXSBcX776rNcTjSOqDxZ5SDty-4/edit?usp=sharing)和冗长的[文本](https://docs.google.com/document/d/1SxmQBrDZvj16veuc2OVO0wUX7a7vEKPM-57dNLXhuEk/edit?usp=sharing)中介绍了我们的工程阶梯，包括所有缺点。请随意复制这个阶梯并更新它，以便在您自己的组织中使用。
