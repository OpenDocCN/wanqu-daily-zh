# 如何在软件开发中放慢速度以达到前所未有的速度

### 关键要点

*   匆忙既不会让我们更快，也不会让我们更有效率；这会增加压力，分散注意力。我们需要创造力、效率和专注。
*   雇佣更好的人才，一起做，一起实践，一起学习，在你的组织中提高专业素养，培养工匠精神。
*   通过制定计划和经常修改计划、收集和分析以及消除浪费来提高团队的适应性和流程的效率。
*   没有高质量的代码库，你就不可能敏捷。把缺陷推下去，频繁发布，先测试再重构，关注简单的设计。
*   工作软件不一定要制作精良。只有优秀的专业人员才能构建出精心制作的软件，只有精心制作的软件才能让你比以往更快地构建。

不受控制地快速前进可能是软件开发的最大敌人。你应该放慢脚步的三个主要领域是人员、流程和产品。在深究细节之前，我先说一个故事。

我猜是 2011 年。我加入了一个负责搭建网络营销平台的团队。我的主要职责是尽可能快地给系统添加新功能。嘿，我是高级开发员。当开发人员能够比其他人开发得更快时，我们称他们为“高级”，对吗？然而，当我加入时，我们注意到由于技术债务和设计挑战，几乎不可能走得很快。在每一次提高速度的尝试中，我们都注意到我们增加了复杂性并破坏了质量。在我看来，加快速度的唯一方法是从头开始重写整个系统。

我记得，我打电话给产品经理，说我们需要重写整个系统。电话那头沉默了 30 秒后，项目经理回答道:“你是说你的团队写的产品质量太差，以至于同一个团队不得不重新编写同样的产品，但这次要更好。对吗？抱歉伙计，这是不可接受的。你应该写得更好。”

#### 相关赞助内容

根据 [Standish Group 的混沌报告](https://www.projectsmart.co.uk/white-papers/chaos-report.pdf)，94%的软件项目都是从零开始重新开发不止一次。那是巨大的。同样，当我看我过去工作的产品时，我看到几乎所有的产品都是用更新的技术、架构和设计从零开始重写的。重写在这个领域是如此普遍，以至于企业公司经常把它看作是项目管理和创新的唯一选择。我们写，重写，重写，一遍又一遍。

我们必须了解我们在软件开发中最大的敌人。在软件世界中，速度是至关重要的。不仅成为市场上的第一人很重要，而且通过添加新功能和快速消除错误来响应客户也能保持较高的客户满意度。但是我们都有一个“速度”的问题。我们假设走得更快、更聪明、更有效率与给目标设定最后期限有关。我们认为，通过更多的工作或与更多的人一起工作，我们可以走得更快。因此，我们要么增加新员工，要么加班加点加快生产。匆忙既不会让我们更快，也不会让我们更有效率。匆忙会增加压力，分散注意力，破坏生产力。相反，我们需要创造力、效率和专注。

软件开发非常困难和复杂。我们无法摆脱复杂性。因此我们不得不忍受它。对速度的需求创造了一个不稳定、不可持续的环境，让我们压力大、注意力不集中、效率低。就是不行。团队能力、总体规划、估算、固定工作时间、期限和速度概念都是虚构的；无能是现实。交付时间直接取决于人员技能、流程效率和产出质量。很多时候，开发者给自己隐藏的截止日期，没有任何真正的需要。

最终，我们会得到遗留软件。期限的压力加上无能导致遗留软件，这是一个工作软件的死胡同。一段时间以来，我一直用一个不同的术语来形容遗留软件:僵尸软件。僵尸软件更适合，因为这种软件实际上已经死了，但在生产中似乎还活着。它在生产中起作用，人们从中赚钱，但它需要软件开发人员的鲜血、生命和精力来以某种方式继续工作。开发人员不敢碰它，因此如果它能工作，没有人想改变它。

Robert C. Martin 在 twitter 上有一句关于遗留软件症状的完美名言:“如果你的软件越来越难开发，那你就是在做错事。”在狂奔的同时，我们也在如此破坏质量，以至于我们前进的每一步都让整个进度和流动比以前慢了。我认为放慢速度，直到我们达到一个可持续的页面，这是唯一可以走得更快的方法。

## 在软件开发中，匆忙是邪恶的

正如 Robert C. Martin 在 CleanCoders 提到的软件的主要价值，“软件系统容忍和促进这种持续变化的能力是软件的主要价值”。在软件开发中，匆忙是邪恶的。任何匆忙的尝试都会对软件的生产力、专注度、人员效率、适应能力和容忍度造成巨大的损害。

例如，我们总是有时间修复 bug，却没有时间编写测试。我们没有重构和编写测试，因为我们没有足够的时间。但是我们有时间调试、破解代码和修复错误。

我们如此关注过程，以至于我们经常忘记软件开发中的主要资产:人。过程帮助人们改进他们制造产品的方式，增加动力和培养健康的环境。最后，流程的效率很重要，但人是关键。

我们必须承认没有人和事是完美的。客户、老板、经理、团队成员、商务人士，甚至你自己，都远非完美。需求、文档、工具、代码、你构建的系统，以及设计，也不可能是完美的。所以，我们不得不停止不加控制的狂奔和提速。以可持续的速度前进的唯一方法是放慢速度，这有三个重要方面:

*   提高专业精神和工艺的人
*   提高适应性和效率的过程
*   提高自动化和质量的产品

## 当涉及到人时，需要放慢速度的领域

过程和工具不能构建产品，但是人可以。我们不得不承认，“人才招聘”是一个组织最重要的功能。它对公司和产品本身的未来有直接的影响。

为你的组织雇佣最优秀的人才。我说的“最好”并不是指周围最聪明或最有经验的人。我最低限度地寻找激情、纪律和动力。如果这三种技能都存在于一个天赋中，其他的技能也可以很容易的发展。招聘是一个双赢的过程，所以双方都应该从这个过程中获益。所以你应该放慢你的招聘进程，并投资改善它。人们加入他们相信的公司。所以模拟你想看到的行为。并通过你的公司文化，你的愿景和人，让人才相信你。

自我是一种氰化物，会慢慢杀死你的组织。永远不要让自我进入你的组织。从可爱的傻瓜到天才怪人，永远不要让极端分子加入你的团队。永远不要雇佣太自负的人。和这些人在一起，你永远也不可能建立一个让人钦佩的公司文化。

停止单独工作，开始合作。永远不要让孤岛发生，因为孤岛或英雄开发人员是功能失调的组织的症状。紧紧地坐在一起。一起定义团队标准。两人一组和小怪一起工作；一起复习。让责任在团队中分担。

一起练习是提高技能最有效的方法。在合作的过程中，我们不仅激励了人们，还相互学习。在你的团队中定期组织代码务虚会、随机会议和编码会议。每个工作日花 30 分钟练习。

让知识在人与人之间流动。一起学习。自 2010 年以来，我一直在我工作的团队中每周组织棕色袋子/午餐和学习会议。在不同的时间，我两次从我的同事那里听到，“每周三参加会议让我提高了自己，这激励了我很多”。这反映了公司定期内部会议的力量和影响。

收集和传递反馈。为了收集集体反馈，你可以像我多年来所做的那样组织大型回顾会。对了，大回顾是一种新型的挖掘问题的回顾，20 人以上。

教学和分享是掌握一个话题的最好方法。做一个演说家，回馈社会。

开发人员似乎讨厌文档，但实际上恰恰相反。人们阅读的任何输出都只是一个文档；从生产代码本身到测试代码，从提交消息到提交图，从日志消息到错误消息，开发人员无意中记录了很多。所以无论你记录什么，因为人们阅读它是为了理解，所以做得更好。

你不是孩子。公司不是你的父母。我们必须拥有你的职业生涯，投资于你自己。如果投资意味着花费时间和金钱，那就为自己做吧。

## 我们如何通过放慢速度来优化流程？

每一天，我们都面临新的挑战。这些挑战不应该仅仅是市场需求或新的要求。技术挑战对我们的进步也有很大影响。

计划什么都不是，但是计划就是一切。一定要经常计划和修改。尤其是在创业的早期阶段，你需要极度的敏捷。每天一次对齐，比如通过每日 Scrum 或每日 standups，是不够的。你必须紧密合作，两人一组，每天不止一次地保持一致。保持您的迭代长度较短，短至一周。通过组织定期评审和演示会议，创建多反馈循环渠道。

定义短期目标和长期目标。短期目标为你的团队创造焦点，长期目标防止分散注意力。

如果你想知道你错在哪里，从可视化流程开始，包括技术和商业方面。想象失败和混乱，从过去的经历中学习。

永远不要凭直觉做决定。始终收集数据，分析并根据数据做出决策。允许每个开发人员访问产品和代码指标也很重要。这增加了产品开发的集体所有权和常识。

废物是你生产的任何没有商业价值的东西。检测并消除办公室、代码和流程中的浪费。童子军离开营地时比发现时干净。同样的哲学也适用于软件开发。遵循童子军规则，让你的代码更干净。当您打开文件以添加新功能并注意到其中的问题时，无需获得任何权限即可修复它。不要忘记在修复问题之前编写测试。这让你在触摸你的代码时感到自信和舒适。

您可以在软件开发生命周期的每一点发现浪费。遵从你对完成的定义，消除“完成 90%，剩余 90%以上”的任务。绝不允许长寿命的分支。长寿的树枝被认为是邪恶的。不要通过手工测试来验证你的代码。手动测试主要验证快乐路径。所有其他场景只能通过测试代码来验证。所以要认真对待。

## 放慢速度如何提高产品质量？

有一点是清楚的。抱歉，没有高质量的代码库，你就不可能敏捷。你需要做的第一件事就是消除技术债务，解决 bug。如果您需要暂时停止构建特性，并专注于消除 bug。

“修复 bug，然后部署到服务器”在今天不是一个合适的过程。它包含风险和危险。我们需要一种更好、更有条理的方法来做这件事。当您想要修复一个 bug 时，首先编写一个测试并以编程方式重现问题。然后修复 bug，看看测试是否通过。之后部署到生产环境是安全的。

我所在的团队几乎把所有的时间都花在了修复 bug 和维护代码库上。这些团队饱受生产不稳定之苦。为了在修复错误的同时继续开发新功能，您需要将您的团队分成虚拟团队。例如，我们在每次迭代中选择两个队友来交付直接的技术支持并继续修复 bug。我们称他们为蝙蝠侠和罗宾。不管你抢的是什么样的功能，bug 都得不折不扣的修复。

今天，开发人员通常使用一种方法来减慢进度，以便加快速度。也就是使用拉请求。他们停止生产线，进行验证和代码审查，以提高代码质量。他们从不将未经审查的代码部署到产品中。

我们的最终目标应该是实现持续交付，并频繁发布。从 git 分支机制到部署策略，从反馈机制到自动化测试实践，都需要不同的思维模式。

您在 SDLC 中使用的实践表明您的开发速度有多快。对于 git 分支机制，哲学“尽早提交，经常提交，稍后完善，发布一次”和基于主干的开发以及功能切换让您消除浪费。

我用 TDD 已经很多年了。很多人向我抱怨 TDD 对编程速度的影响。 [Joe Rainsberger 在 twitter](https://twitter.com/jbrains/status/167297606698008576) 上分享了他对 TDD 的想法:“担心 TDD 会拖慢你的程序员？不要。他们可能需要慢下来。”

TDD 的重构多于测试，思考多于编码，简单多于优雅。这就是为什么它导致更好的质量。通过 TDD 开发；有足够的测试和简单的设计。

你曾经达到过 100%的代码覆盖率吗？我在一个为期三个月的项目中实现了这一点。我对每一行产品代码都编写了单元测试。那时候，我觉得自己是一个拥有超能力的英雄。但是，当我们部署到 UAT 的生产前环境时，我们注意到有四个特性无法工作。我必须编写集成和功能测试来检测错误并解决它们。然后我意识到单元测试并不能保证良好的设计和工作功能。停止计算代码覆盖率。代码覆盖率只不过是愚蠢的信息辐射器。

如果您需要 30 分钟或更少的时间来修复错误，请修复它们。此外，用 20%的时间消除技术债务。

我们通常编写代码是为了将来不改变它。因此，当我们设计软件时，类似地，我们选择技术和工具，以便将来不会改变它们。但是我们错了。重构应该出现在我们开发过程的每个阶段。正如肯特·贝克所说，我们必须做简单的改变，让改变变得简单。为了实现这一目标，我们在一个 mono 存储库中管理我们所有的微服务。Mono repository 是为了让重构变得简单，这是我们真正需要的。

任何在被要求之前采取的设计决策都是错误的。因此，我们等到最有责任感的时候再采取行动。我们使用六边形架构在系统的高层次设计上主动松散耦合和高内聚。这也导致了精心设计的独石。顺便说一句，独石并不邪恶，糟糕的设计才邪恶。我们总是从一个整体开始，在端口和适配器的帮助下，我们提取一些功能作为微服务。正如 Martin Fowler 在他的博客的“Monolith First”文章中所说的 [，“直接采用微服务架构是有风险的，所以首先考虑一个 monolithic 系统。如果您需要，可以随时拆分为微服务。”](https://martinfowler.com/bliki/MonolithFirst.html)

## 放慢速度以加快速度是一种哲学

安德烈亚斯·穆勒在一条推文中提到了他对软件开发的感受:“我不想写只能工作的代码:我想写干净、可维护、易于理解和测试良好的代码。”

为了实现这一目标，我们需要关注三个方面:人员、流程和产品。通过放慢人们的脚步，我们的目标是提高专业水平和工艺水平。通过放慢流程，我们旨在提高适应性和效率。通过放慢生产速度，我们的目标是提高自动化和质量。当我们关注这些领域时，我们就开始培养一种能够快速开发软件的开发文化。

我们不应该忘记以下几点。工作软件不一定要精心制作。只有优秀的专业人士才能开发出制作精良的软件。只有精心制作的软件才能让你比以往更快地构建功能。
