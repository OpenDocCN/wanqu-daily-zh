# id 软件的编程原理

## id software 是如何在不到 6 年的时间里用不到 10 名开发人员推出 28 款游戏的

id Software 可能是 20 世纪 90 年代最具标志性的视频游戏公司，开发了诸如 Wolfenstein 3D、Doom 和 Quake 等开创性的游戏。在最近的一次[演讲](https://www.youtube.com/watch?v=KFziBfvAFnM)，[约翰·罗梅洛](https://en.wikipedia.org/wiki/John_Romero)(id 的联合创始人)，概述了该公司的编程原则，这些原则让他们在很短的时间内，用一个很小的团队制作出如此多的高质量游戏。

可以把这看作是兰普森关于系统设计的永恒论文的伴侣。许多原则都是通用的，它们在今天都非常适用。



That’s one new game every 2.35 months. Whoa.



虽然这些编程原则指的是视频游戏开发，但大多数(如果不是全部)适用于一般的软件开发。在这篇博文中，我概述了自己对这些原则的解读，并配上了罗梅罗自己幻灯片中的截图以供参考。我还试图将每个原则推广到游戏开发之外。开始了。

# **原则 1:只管去做(并且做好)。**

这并不一定意味着过度复杂化你当前版本的产品。事实上，这可能是描述迭代开发实践的一种方式。做一些能真正做好一件事的东西，并在此基础上不断改进。只要在每次迭代中保持高质量标准。



# 原则 2:保持你的代码总是可运行的。

在他的演讲中，罗梅罗提到了他们如何编写代码，当用户加载雪碧出错时，显示一个百吉饼的图像。通过添加良好的默认/回退，他们使游戏仍然可以玩。如果他们没有这样做，用户将被阻止，直到 bug 被修复(即损失了工作时间)。你可以想象当一个工程团队变得越来越大时，这变得多么重要。一个实际的例子是在 ReactJS 中使用 [defaultProps](https://reactjs.org/docs/react-component.html#defaultprops) 。



# 原则 3:保持简单。

这显然不新颖。称之为[奥卡姆剃刀](https://en.wikipedia.org/wiki/Occam%27s_razor)或[接吻原则](https://en.wikipedia.org/wiki/KISS_principle)，让事情尽可能简单是永恒的好建议。



# 原则 4:投入时间构建伟大的工具。

在他的演讲中，罗梅罗提到他建立了一个名为 TED 的关卡编辑器。他花在打造 TED 上的时间获得了丰厚的回报，因为这极大地提高了生产率，帮助他们迅速推出一个又一个游戏。从那时起，帮助提高开发人员生产力的开发工具就开始激增。但是如果一些现成的东西不能满足它，试着确定一个内部工具是否能帮助你的开发人员达到最大的生产力(即使它能从主产品中取走开发资源)。



# 原则 5:彻底测试你的代码。

这涵盖了许多最有效的工程团队作为最佳实践使用的许多主题:(a) [狗粮](https://en.wikipedia.org/wiki/Eating_your_own_dog_food)你的产品尽可能多；(b)不要委托其他人(例如质量保证工程师，或者更糟，客户)去发现你代码中的错误；(c)编写尽可能多的代码测试。



# 原则 6:尽快修复 bug。

我们在 [AgentRisk](https://agentrisk.com) 对这一点非常严格，这是我们从[以前的](https://en.wikipedia.org/wiki/Bugsense) [创业公司](https://www.crunchbase.com/organization/maginatics)沿袭下来的做法。在我们每天的站立会议中，我们确保任何新的错误都有最高优先级，并尽快得到修复。显然，并非所有的 bug 都是相同的，因此这里需要一些与业务相关的判断。



# 原则 7:使用高级开发系统。

这可能是最适用于游戏开发的一个。在其他情况下，当涉及到开发期间的测试时，您可能想走另一条路。例如，您可能让用户在一个性能非常差的移动设备上运行您的应用程序，或者他们可能通过高延迟的 2G 连接访问您的 web 应用程序。确保他们没有一个糟糕的 UX。



# 原则 8:为这个版本的产品编写代码。

这通常解释为“不要将过去代码及其实现的限制转移到未来的代码中”。这是一个棘手的问题，与原则 4 有点关系。作为工程师，我们经常被诱惑从头开始“重写一切”。这就是拥有经验丰富的工程领导真正重要的地方。通常，选择冒险进行新的实现会带来收益(类似于构建工具)。例如，随着 Slack 开始扩展，[他们完全放弃了他们的 1.0 版本实现](https://softwareengineeringdaily.com/2020/02/27/slack-frontend-architecture-with-anuj-nair/)并转移到一个全新的架构，收获了数倍的回报。我们也有过从 Python 到 Elixir 的类似经历，拥有更健壮的代码库和更高的开发效率。



# 原则 9:使用好的组件抽象。

这真的很难。如果你曾经构建和维护过一个 API，你就会知道做好它有多难(尤其是第一次)。在他的演讲中，罗梅罗举了一个例子，将火炬的功能与火焰和其他相关物体封装在一起。如果他们需要移动或更新一个级别中的所有火炬实例，更细粒度的抽象可能会导致例如忘记移动/更新火焰。在这上面花很多时间，尝试第一次就做对。在开发和调试期间会有复合回报。



# 原则 10:编码时寻求同事的反馈。

使用代码审查软件可以对此有所帮助。对于产品中更复杂的部分，可能需要进行高级架构评审。无论如何，确保你提倡一种重视沟通和寻求反馈的文化。



# 原则 11:给编码人员创造的自由。

切洋葱有很多方法。给你的程序员创造的自由，让他们想出自己的解决方案来解决他们正在处理的问题。只要确保执行*一些*编码标准，这样团队的任何成员都可以进入代码库。沉迷于编码美学会浪费宝贵的时间，所以最好把这留给 linters 和自动格式化程序。这并不意味着不应该鼓励在代码评审中识别次优的实现。只关注客观错误的东西。
