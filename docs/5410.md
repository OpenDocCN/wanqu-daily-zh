# 什么时候需要决策框架？

公司的绝大多数决策都是低风险的，应该由该领域的负责人单方面做出(例如，我们是否应该将本周的例会从周一改到周二)。

只有当高风险决策缺乏明确性时，才需要决策框架。更高的风险可能意味着决策具有长期影响，或者如果做出了错误的决策，平仓的成本可能会很高。

这个过程可以是相当轻量级的(我见过它从开始到结束不到 15 分钟，在会议期间直播)，但它也可以用于数周的更大决策。

有了经验，你可能会提前知道框架什么时候是有益的。其他时候，只有当你注意到对下一步有分歧或不清楚时，它才会变得清晰。

不管是哪种情况，一旦你决定需要这个框架，你就可以按照下面的步骤来做。

# 比特币基地决策框架

# 1.设置参数

在这一步，你会让每个人都知道如何做出决定。这有助于提前获得认同，并减少后续的第二次猜测。



Setting the parameters of a decision up front



你会注意到这里的一些片段:

*   **决策日期**
    提前选择一个截止日期有助于避免分析瘫痪(等待更多信息到来的时间过长)。它还为那些被阻止等待这一决定的任何一方提供了保证。
*   **重访日期**
    请团队成员提前就他们对该决策的承诺时间达成一致。这有助于避免过早地重新做出决定。没有什么决定是一成不变的，你只是想要一个关于如何以及何时重新讨论的明确协议。决策者还有权在决策出错时按下“紧急按钮”，以便更快地重新审视决策。

参与任何决策的有三类人。

*   **决策者** —做出决定的人。
*   **输入提供者** —影响决策的人。
*   **受影响方** —受决策影响的人。

输入提供者的理想数量是三到八个人。超过这一点，小组讨论的回报就会递减。决策可能会影响到更广泛的群体(受影响的各方)，所以尽量让每个受影响的各方至少由一个输入提供者来代表。

最后，有三种类型的决策:

*   二元决策 —是/否决策，如雇佣某人或收购一家公司。
*   **优先排序** —强制对多个选项进行排序，例如从一百个可能的选项中选择接下来的五个功能进行构建。
*   **选择** —在许多可能性中选择一个选项，例如新产品的名称。

## 应该有一个还是多个决策者？

对于大多数决定，我喜欢选择一个决策者来保持简单。然而，让多个决策者来做高风险决策是有意义的(一个*错误的是*决策的成本大于一个*错误的否*决策的成本)。

就像一家银行可能需要多次批准才能转移大量资金一样，您可以通过增加更多决策者并赋予每个决策者否决权来对高风险决策进行“控制”。这意味着只有当每个决策者都是“是”的时候，一个“是”的决定才会发生。

如果你一直要求每个输入提供者一致同意，这将进一步减少潜在的负面影响，但也会限制你的正面影响(“委员会设计”)。是的，在某些领域(如风险投资)的决策可能看起来像是一大群人不太可能达成一致的反向押注，因此过于追求安全可能是危险的。

权衡这些利弊，为你的特定决策找到合适的决策者数量。



一般来说，如果一个错误决策的成本是可以忍受的，那么选择一个决策者来获得速度红利。如果一个错误的决定是不可逆转的或者代价太大，增加更多的决策者。

# 2.故意的

接下来，你需要让所有的输入提供者(和决策者)在一个房间里分享信息，并获得他们的投票。



Sample deliberation step where you capture votes, and pros/cons of each option.



*   **列举选项**
    在这里做一个迷你头脑风暴的选项。先追求数量，然后再做判断。让每个人都创造性地思考。完成后，看看是否可以合并任何重复的选项。注意:如果你正在做二元决策，你可以跳过这一步，因为你已经知道选项:是和否。
*   **展示数据**
    如果有任何从客户、对比测试、法律团队、外部顾问等收集的数据，现在是向团队展示的时候了。理想情况下，在会议讨论之前，这些内容会作为预读材料共享。
*   **第一轮投票(盲投)**
    捕捉第一轮投票(来自输入提供商)以测量房间的温度，最好是没有任何人提前看到其他人的投票。对于一个*二进制*的决定，让每个人在房间里数到三时竖起大拇指或放下大拇指。对于*优先顺序*或*选择*决策，让每个人在电子表格中投票(鼓励他们不要看其他栏)。
*   **讨论利弊**
    在房间里走来走去，从级别最低的到级别最高的，让每个人都分享他们投票的原因。允许自由但有时间限制的讨论。记录每个选项的备注字段中讨论的优点和缺点。鼓励每个人畅所欲言，保持好奇心，学习新的东西。在这个阶段，你会经常发现你列举的选项之间存在灰色阴影，没有明确的“最佳”选项。您可以继续更新列表，添加或合并选项。
*   **投票第二轮**
    最后，在每个输入提供者听到其他人的想法(完全关联的信息共享)后，要求进行第二轮投票，并查看是否有人的投票发生了变化。将第二轮投票记录在电子表格中以备记录。

偶尔在这一步你会在房间里有明确的共识，决策者的工作很容易。他们可以简单地交流他们的决定，这个过程就完成了。但更常见的是，决策者会进入最后一步。

如有必要，您可以在决策日期的限制内完成额外的讨论和投票。

# 3.决定



Memorialize the decision for later view.



*   **做决定**
    决策者现在可以花一两天的时间仔细考虑输入的信息，并做出最终决定。即使没有明确的“最佳”选择，他们仍然需要在决策日期前做出决定。
*   传达它
    用电子邮件把决定发给所有受影响的人。包括输入提供者列表、考虑的选项、简要的理由(赞成和反对)以及回访日期。你不需要包括谁投了哪一票，因为在这一点上你们都是作为一个统一的团队前进的。
*   **记住决定**
    最后，将电子表格保存在一个永久的地方(比如谷歌文档)，以便日后查阅。您可能希望在重访日期安排一个提醒，以将输入提供者拉回到一起，作为一个学习练习。回顾错误的决定是如何做出的(事后检讨)和防范历史修正主义者(我知道这没用)是很有趣的。).

如果决定是有争议的，通过电子邮件发送会给人们时间来异步处理，而不是在房间里得到他们的第一反应。

# 结论

在许多组织中，困难的决策可能是压力的来源，但它们不一定是。在高绩效的组织中，它们是取得快速进步、创新和成功的机会。

有很多方法会把决策搞砸，就像任何事情一样，随着时间的推移，你会做得更好。我在下面列出了一些故障模式，并试图给它们起一个好记的名字:

*   **喋喋不休**
    一堆一次性的对话发生了，而不是让所有的输入提供者在一个房间里一起分享知识(完全连接的信息共享)。
*   **蝇王** 在战线划定之前，没有明确的决策者选定。
*   **最后一分钟添加** 意识到有人应该在流程的后期成为输入提供者，并试图让他们溜进来。
*   利益冲突
    有偏见的决策者被选中，而不是中立的。
*   **在杂草中**
    一位经理选择他们自己作为决策者，而这本来可以委托给他们团队中的某个人。
*   一位经理选择其他人作为决策者，然后在最后一刻否决他们！(如果你要授权，你必须真心同意决策者的任何决定)
*   **唯唯诺诺的人**
    在听取别人的想法之前，让房间里最高级/最有特权的声音左右谈话。
*   **偶然的是**
    讨论发生在电子邮件(产生+1 的势头)、一次性谈话或与大忙人的紧急会议上。似乎达成了共识，但是输入提供者没有对这个决定给予应有的考虑。
*   **分析瘫痪**
    决策被推迟到决策日期之后，希望得到更多的数据。(“我们来拆分测试一下吧！”如果走得太远，可能会是这种形式。)
*   创造力杀手
    未能将选项的列举与选项的评估分开。
*   **撕掉创可贴**
    让“没有好的选择！”阻止你选择“最不坏”的选项。
*   **流程太多** 在经理本可以单方面、更快地做出低风险决策时，使用决策流程。
*   **无意的民主**
    决策者感到压力，不得不选择票数最多的选项，放弃选择他们认为真正最佳的选项的责任。
